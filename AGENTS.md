# AGENTS.md

## 1. Назначение файла

Этот файл определяет правила работы AI-агента с репозиторием `Programming-Digital-Systems`.

AI-агент должен использовать этот файл перед созданием, изменением или анализом документов проекта.

Главная задача агента — сохранять масштаб, связность, структуру и стиль базы знаний по проектированию и разработке цифровых систем.

## 2. Масштаб проекта

Проект не является набором отдельных заметок.

Проект развивается как универсальная связанная база знаний по проектированию, разработке, документированию, проверке, сопровождению и развитию цифровых систем.

Проект должен быть пригоден для следующих форматов:

- учебный курс;
- инженерный справочник;
- система roadmap-документов;
- система проектных анкет;
- энциклопедия понятий;
- книга;
- серия книг;
- Obsidian-база;
- GitHub-документация;
- методология проектирования цифровых систем.

## 3. Обязательные документы для чтения перед работой

Перед созданием или редактированием документов AI-агент должен прочитать следующие документы:

1. `PROJECT_SCOPE.md`
   - Определяет масштаб проекта, центральную формулу цифровой системы и области применения.

2. `docs/00_maps/00_00_Documentation_Map.md`
   - Определяет карту базы знаний, слои документации и маршрут движения пользователя.

3. `docs/01_regulations/Documentation_System_Regulation.md`
   - Определяет общие правила построения системы документации.

4. `docs/01_regulations/Document_Writing_Rules.md`
   - Определяет стиль изложения, запреты, проверяемость и правила отделения категорий от примеров.

5. `docs/01_regulations/Link_Rules.md`
   - Определяет правила ссылок между документами, разделами, смысловыми блоками, цитатами и диаграммами.

6. `docs/01_regulations/Diagram_Rules.md`
   - Определяет правила выбора и оформления диаграмм.

7. `docs/02_templates/Roadmap_Document_Template.md`
   - Используется перед созданием roadmap-документов.

8. `docs/02_templates/Questionnaire_Document_Template.md`
   - Используется перед созданием анкет.

## 4. Порядок чтения

Перед работой агент должен читать документы в следующем порядке:

```text
PROJECT_SCOPE.md
↓
docs/00_maps/00_00_Documentation_Map.md
↓
docs/01_regulations/Documentation_System_Regulation.md
↓
docs/01_regulations/Document_Writing_Rules.md
↓
docs/01_regulations/Link_Rules.md
↓
docs/01_regulations/Diagram_Rules.md
↓
соответствующий шаблон документа
```

Если создаётся roadmap-документ, агент должен дополнительно читать:

```text
docs/02_templates/Roadmap_Document_Template.md
```

Если создаётся анкета, агент должен дополнительно читать:

```text
docs/02_templates/Questionnaire_Document_Template.md
```

## 5. Главные правила создания документов

AI-агент должен соблюдать следующие правила:

- документ должен быть частью общей системы знаний;
- документ должен иметь назначение;
- документ должен показывать своё место в маршруте разработки;
- документ должен иметь связи с другими документами, если такие связи существуют;
- документ должен различать входные и выходные документы;
- документ должен содержать проверяемые формулировки;
- документ не должен содержать личный шум;
- документ не должен смешивать требования, инструменты и реализацию;
- документ не должен смешивать категории и примеры на одном уровне;
- документ должен использовать диаграммы, если информация требует визуального представления;
- документ должен быть пригоден для расширения до уровня книги, серии книг или энциклопедии.

## 6. Запрещено

AI-агенту запрещено:

- создавать изолированные документы без места в общей системе;
- создавать документы без раздела назначения;
- создавать roadmap-документы без входных условий и критериев завершения;
- создавать анкеты без критериев заполнения вопросов;
- добавлять случайные ссылки без объяснения смысла связи;
- использовать декоративный текст;
- использовать эмоциональные оценки;
- писать свободные рассуждения вместо правил;
- смешивать виды документов с примерами этих видов;
- выбирать инструменты реализации внутри технических требований, если это не внешний обязательный constraint;
- добавлять диаграммы только ради оформления.

## 7. Правило ссылок

Если документ связан с другим документом, ссылка должна объяснять смысл связи.

Неправильно:

```md
- `01_Roadmap_System_Design.md`
```

Правильно:

```md
- `01_Roadmap_System_Design.md`
  - Передаёт: сущности, данные, правила, состояния, события, потоки, хранение и ошибки.
  - Используется для: формирования технических требований.
  - Ограничение: не должен содержать выбор конкретных библиотек и фреймворков.
```

## 8. Правило диаграмм

Если информация описывает структуру, связь, поток, состояние, последовательность или иерархию, агент должен рассмотреть необходимость диаграммы.

Тип диаграммы должен соответствовать смыслу информации:

- маршрут или алгоритм — `flowchart`;
- сущности и связи — `classDiagram`;
- состояния — `stateDiagram`;
- последовательность взаимодействий — `sequenceDiagram`;
- иерархия понятий — `mindmap` или `flowchart`;
- архитектурные уровни — C4 или совместимая flowchart-адаптация.

Если документ предназначен для Obsidian, Mermaid-синтаксис должен быть максимально совместимым.

## 9. Правило анкет

Анкета должна вести пользователя от неопределённой идеи к проектному решению.

Каждый вопрос анкеты должен иметь:

- идентификатор;
- формулировку вопроса;
- пояснение;
- поле для ответа;
- критерий заполнения;
- связанные документы;
- следующий шаг.

## 10. Правило примеров

Если правило универсальное, примеры должны показывать применение в разных областях цифровых систем.

Категории примеров:

- скрипты автоматизации;
- GUI-приложения;
- web-системы;
- embedded-системы;
- PLC-системы;
- CNC/CAM-системы;
- базы данных;
- интеграционные системы.

Примеры должны быть вложены внутрь категорий, а не стоять на одном уровне с категориями.

## 11. Правило изменения структуры

Если агент создаёт новый крупный документ, слой или маршрут, он должен проверить необходимость обновления:

- `docs/00_maps/00_00_Documentation_Map.md`;
- связанных roadmap-документов;
- связанных анкет;
- связанных регламентов;
- связанных диаграмм.

## 12. Правило результата

Результат работы должен быть пригоден для практического использования.

Документ считается плохим, если он выглядит написанным, но не помогает пользователю двигаться дальше по проектному маршруту.

Документ считается хорошим, если он:

- объясняет место этапа;
- показывает вход и выход;
- содержит проверяемые правила;
- связан с другими документами;
- содержит нужные визуальные элементы;
- помогает пользователю принять следующее проектное решение.

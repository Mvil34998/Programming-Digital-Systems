# Metamodeling Methods and Standards

## 1. Назначение документа

`01_Metamodeling_Methods_And_Standards.md` фиксирует исследовательские ориентиры для проекта [[Digital_System_CAD_Concept_for_Codex|Digital System CAD]].

Документ нужен Codex и пользователю для понимания методов, стандартов и правил метамоделирования, которые могут помочь сформировать рабочую метамодель цифровой системы.

> [!info] Главное
> Текущий этап проекта — не разработка metamodeling workbench и не реализация приложения. Текущий этап — изучение методов метамоделирования и формирование рабочей формы метамодели цифровой системы.

## 2. Текущий статус

На данном этапе проект не строит:

- редактор диаграмм;
- генератор SDD;
- metamodeling workbench;
- приложение для моделирования;
- замену Archi, UML, SysML, EMF или MPS.

Проект сейчас собирает и формирует информацию, чтобы понять:

- какие методы метамоделирования применимы к цифровым системам;
- какие понятия цифрового мира можно считать элементами модели;
- какие связи между элементами нужны;
- какие проверки и ограничения должны существовать;
- как из модели могут получаться анкеты, таблицы, диаграммы, SDD, задачи, тесты и контекст для Codex;
- где заканчивается универсальное ядро и начинаются доменные расширения.

## 3. Важное различие

| Понятие | Значение |
|---|---|
| Model | Описание конкретной цифровой системы |
| Metamodel | Правила построения моделей |
| Meta-metamodel | Правила описания метамоделей |
| Modeling language | Язык моделирования, например UML, BPMN, SysML, ArchiMate |
| Metamodeling method | Способ выделения элементов, связей, правил, ограничений и представлений |
| Metamodeling workbench | Возможная будущая среда для создания языков, моделей, редакторов и генераторов |
| View | Представление данных модели: диаграмма, таблица, форма, документ |
| Transformation | Преобразование модели в другой вид: SDD, задачи, тесты, кодовый контекст |

Главное правило:

> Metamodeling workbench — это возможное будущее направление реализации, а не текущая цель работ.

## 4. Как использовать стандарты

Стандарты и технологии из этого документа не являются прямым техническим заданием.

Они используются как инженерные ориентиры:

- чтобы не смешивать модель, метамодель и представления;
- чтобы не принимать диаграмму за источник правды;
- чтобы отделять данные модели от документации;
- чтобы формулировать проверки модели;
- чтобы проектировать трассировку;
- чтобы понимать, как модель может превращаться в SDD, задачи, тесты и контекст для Codex.

## 5. Reference Standards

### 5.1. MOF / Meta Object Facility

MOF полезен как ориентир для уровней метамоделирования:

```text
M3 -> meta-metamodel
M2 -> metamodel
M1 -> model
M0 -> real system / runtime facts
```

Вывод для проекта:

> Digital System CAD не должен реализовывать MOF сейчас, но должен учитывать разделение уровней M3 / M2 / M1 / M0.

### 5.2. MDA / Model Driven Architecture

MDA полезна как ориентир для перехода:

```text
platform-independent model -> platform-specific model -> implementation
```

Вывод для проекта:

> Сначала нужно описывать цифровую систему независимо от конкретного инструментария, а уже затем связывать модель с реализацией.

### 5.3. XMI / XML Metadata Interchange

XMI полезен как пример машинного обмена моделями.

Вывод для проекта:

> На раннем этапе можно использовать Markdown, YAML, JSON или таблицы. Но границы импорта и экспорта нужно проектировать так, чтобы модель можно было переносить между представлениями.

### 5.4. OCL / Object Constraint Language

OCL полезен как ориентир для правил проверки модели.

Примеры проверок для Digital System CAD:

- каждое функциональное требование должно быть связано хотя бы с одним TestCase;
- каждый DataField должен принадлежать Entity, Interface или Integration;
- каждое Rule должно иметь условие, действие и поведение при нарушении;
- каждая Task должна иметь связь с Requirement, Module, Service, Rule или TestCase;
- каждый CodeArtifact должен быть связан с Service, Module, Task или TestCase.

Вывод для проекта:

> Нам нужны правила валидации модели. Сейчас они могут быть описаны текстом, но должны мыслиться как будущие машинно-проверяемые ограничения.

### 5.5. QVT / Query View Transformation

QVT полезен как ориентир для преобразований модели.

В Digital System CAD преобразования нужны для:

- model -> SDD;
- model -> diagrams;
- model -> tables;
- model -> task list;
- model -> test plan;
- model -> Codex context.

Вывод для проекта:

> Генерация должна рассматриваться как преобразование структурированной модели, а не как свободный текстовый рендеринг.

### 5.6. ISO/IEC/IEEE 42010

ISO 42010 полезен для понимания architecture description через:

- stakeholders;
- concerns;
- viewpoints;
- views;
- model kinds;
- correspondences;
- correspondence rules.

Вывод для проекта:

> Одного универсального представления цифровой системы недостаточно. Анкета, таблица, диаграмма, SDD-раздел и Codex-context должны рассматриваться как разные views одной модели.

### 5.7. UML

UML полезен как источник понятий для структуры и поведения:

- class;
- interface;
- component;
- package;
- state machine;
- activity;
- interaction;
- artifact.

Вывод для проекта:

> Digital System CAD не должен копировать UML полностью. Нужно брать только те идеи, которые помогают описывать цифровые системы ясно и проверяемо.

### 5.8. BPMN

BPMN полезен для описания процессов:

- task;
- event;
- gateway;
- sequence flow;
- message flow;
- data object;
- error path.

Вывод для проекта:

> BPMN-подход полезен для сценариев, событий, потоков, решений и ошибок.

### 5.9. SysML v2

SysML v2 полезен как современный ориентир model-based systems engineering:

- требования;
- структура;
- поведение;
- анализ;
- проверка;
- трассировка;
- текстовое и графическое представление;
- API для взаимодействия инструментов.

Вывод для проекта:

> Digital System CAD должен учитывать идеи текстовой модели, графических views, трассировки, согласованности и проверки.

## 6. Reference Technologies

### 6.1. EMF / Ecore

EMF/Ecore полезен как практический ориентир:

```text
Metamodel -> Model classes -> Model repository -> Editors -> Validation -> Generators
```

Вывод для проекта:

> Даже если ранняя работа ведётся в Markdown/YAML/JSON, EMF/Ecore помогает понять структуру метамодельной инфраструктуры.

### 6.2. Sirius

Sirius полезен как ориентир для разных views над моделью:

- diagrams;
- tables;
- trees;
- validation rules;
- actions;
- declarative workbench descriptions.

Вывод для проекта:

> Диаграммы, таблицы и деревья должны рассматриваться как views, а не как отдельные источники правды.

### 6.3. Xtext

Xtext полезен как ориентир для текстовых DSL:

- parser;
- linker;
- typechecker;
- compiler / interpreter;
- editor support;
- LSP-compatible tooling.

Вывод для проекта:

> Codex должен работать прежде всего с текстовой или структурированной формой модели: Markdown, YAML, JSON, JSON Schema или DSL-like text.

### 6.4. JetBrains MPS

MPS полезен как пример language workbench и projectional editing.

Он показывает, что одна модель может иметь разные нотации:

- text;
- tables;
- diagrams;
- forms;
- mathematical notation.

Вывод для проекта:

> MPS полезен концептуально: Digital System CAD тоже должен мыслить разные представления как разные способы работы с одной моделью.

## 7. Рабочий вывод для текущего этапа

На текущем этапе проект должен сформировать:

- рабочую форму элемента модели;
- рабочую форму связи;
- рабочую форму реестра;
- рабочую форму view;
- рабочую форму transformation;
- рабочую форму validation rule;
- правила трассировки;
- правила сбора информации через анкеты;
- правила превращения информации в SDD.

Это не финальная метамодель и не реализация workbench.

Это рабочая инженерная форма для сбора, структурирования, проверки и преобразования информации о цифровой системе.

## 8. Правила для Codex

Codex должен:

1. Не предлагать реализацию приложения раньше формирования рабочей метамодели.
2. Не называть текущий этап разработкой metamodeling workbench.
3. Использовать стандарты как ориентиры, а не как обязательство полной реализации.
4. Отличать модель от метамодели, view от модели, generator от документа.
5. Сначала описывать метод и форму работы, затем структуру данных, и только потом возможную реализацию.
6. Предпочитать Codex-readable формы: Markdown, YAML, JSON, таблицы, DSL-like text.
7. Сохранять трассировку между требованиями, элементами модели, правилами, тестами, задачами и кодовыми артефактами.
8. Любое новое понятие сначала помещать в исследовательскую или метамодельную документацию.

# Правила изложения технических документов

## 1. Назначение документа

Документ определяет правила написания технических, учебных и проектных материалов проекта Programming Digital Systems.

Цель правил — исключить личный шум, декоративный текст, нестрогие советы и смешивание разных уровней информации.

## 2. Запрещённые формы изложения

В документах запрещены:

- личные рассуждения;
- эмоциональные оценки;
- декоративные вступления;
- разговорные формулировки;
- нестрогие советы без условия применения;
- повторение одной мысли разными словами;
- формулировки без критерия проверки;
- смешивание требований, инструментов и реализации;
- смешивание категорий и примеров на одном уровне.

## 3. Обязательные формы изложения

Документ должен использовать проверяемые формулировки.

Допустимые формы:

- `Система должна...`
- `Разработчик должен...`
- `Документ должен...`
- `Требование считается выполненным, если...`
- `Если ..., то ...`
- `Запрещено ...`
- `Допускается ..., если ...`
- `Необходимо определить ...`
- `Перед переходом к следующему этапу должно быть зафиксировано ...`

## 4. Правило проверяемости

Каждое требование должно быть проверяемым.

Неправильно:

> Система должна быстро работать.

Правильно:

> Система должна обрабатывать входной файл объёмом до 5000 строк без зависания интерфейса.

## 5. Правило отделения требований от инструментов

Технические требования должны описывать свойства системы, а не способ реализации.

Неправильно:

> Использовать SQLite.

Правильно:

> Система должна хранить данные между запусками в структурированном виде.

Выбор SQLite, PostgreSQL, JSON или другого способа хранения выполняется в Roadmap выбора инструментария.

## 6. Правило отделения категорий от примеров

Категории и примеры нельзя смешивать на одном уровне.

Неправильно:

- Входные данные
- Excel-файл
- PDF-файл
- Конфигурационные данные

Правильно:

- Входные данные
  - Примеры:
    - Excel-файл
    - PDF-файл
- Конфигурационные данные
  - Примеры:
    - JSON-файл настроек
    - `.env`-файл

## 7. Правило примеров из разных областей

Если правило является универсальным, примеры должны показывать его применение в разных областях цифровых систем.

Области примеров:

- скрипт автоматизации;
- GUI-приложение;
- web-система;
- embedded-система;
- PLC-система;
- CNC/CAM-система;
- база данных;
- интеграционная система.

## 8. Правило учебного объяснения

Документ должен обучать образу мышления, но не должен превращаться в свободное эссе.

Учебное объяснение допускается, если оно:

- объясняет причину правила;
- показывает ошибку мышления;
- даёт проверяемый пример;
- ведёт пользователя к следующему проектному действию.

## 9. Правило structured facts для Digital System CAD

Документы, связанные с [[Digital_System_CAD_Concept_for_Codex|Digital System CAD]], должны стремиться описывать систему через структурированные факты, а не только через списки понятий.

Неправильно:

```text
Есть требование.
Есть правило.
Есть тест.
```

Правильно:

```text
Requirement REQ-001 is_verified_by TestCase TEST-001.
Rule RULE-001 validates DataField FIELD-001.
Rule RULE-001 raises Error ERR-001.
Task TASK-001 implements Requirement REQ-001.
```

> [!important] Правило
> Связь между элементами является частью смысла модели. Если связь важна для понимания, проверки, SDD или Codex context, она должна быть описана явно.

## 10. Правило определения терминов

Имя термина не считается определением.

Для важного термина необходимо указать:

- Definition;
- Purpose;
- Context;
- Source;
- Related elements;
- Open questions.

Если термин может иметь несколько значений, документ должен явно зафиксировать нужный контекст.

## 11. Правило завершения документа

Документ считается готовым, если:

- есть назначение;
- определена область применения;
- указаны связанные документы;
- структура не смешивает уровни информации;
- требования проверяемы;
- примеры вложены в категории;
- есть критерии завершения;
- есть следующий шаг.

## 12. Правило содержания нового документа

Новый документ должен отвечать на три группы вопросов.

### 12.1. Вопросы назначения

- Какую задачу закрывает документ?
- Какой тип документа создаётся?
- К какому слою документации он относится?
- Почему существующих документов недостаточно?

### 12.2. Вопросы связности

- Какие документы являются входными?
- Какие документы получают результат?
- Какие понятия связаны с документом?
- Какие диаграммы или чек-листы должны быть связаны с документом?

### 12.3. Вопросы практического результата

- Что пользователь должен получить после работы с документом?
- Какие критерии показывают, что документ заполнен или изучен достаточно?
- Какие открытые вопросы могут блокировать следующий шаг?
- Какой следующий документ должен использовать пользователь?

Если на эти вопросы нельзя ответить, документ должен оставаться черновиком.

## 13. Правило недопущения дубликатов

Перед созданием документа необходимо проверить, нет ли уже документа с тем же назначением.

Дубликатом считается документ, который:

- раскрывает ту же тему;
- относится к тому же слою;
- имеет те же входные и выходные документы;
- не добавляет нового проектного решения, правила, вопроса или примера.

Если тема уже существует, необходимо расширить существующий документ или явно разделить назначение документов.

Пример разделения:

- энциклопедическая статья объясняет понятие;
- roadmap показывает порядок работы;
- анкета собирает ответы;
- пример показывает применение;
- чек-лист проверяет готовность.

## 14. Правило уровня детализации

Уровень детализации должен соответствовать типу документа.

- Roadmap должен описывать порядок работы, входы, выходы и критерии завершения.
- Анкета должна содержать вопросы, поля ответа, критерии заполнения и следующий шаг.
- Энциклопедическая статья должна раскрывать понятие, классификацию, правила и примеры.
- Пример должен показывать применение правила в конкретной области.
- Чек-лист должен проверять готовность результата, а не объяснять тему заново.

Если документ становится слишком большим, необходимо разделить его на:

- основной документ;
- связанные диаграммы;
- примеры;
- чек-лист;
- анкету;

## 15. Правило следующего шага

Каждый документ должен вести пользователя к следующему действию.

Следующий шаг должен быть выражен явно:

- перейти к следующему roadmap-этапу;
- заполнить связанную анкету;
- проверить результат по чек-листу;
- раскрыть связанное понятие в энциклопедии;
- посмотреть пример применения;
- обновить карту или диаграмму;
- зафиксировать открытые вопросы.

Документ не считается завершённым, если после чтения неизвестно, что делать дальше.

## 16. Правило визуального формата документа

Документы проекта должны использовать единый визуальный формат, если они являются учебными, энциклопедическими, roadmap-документами, анкетами, примерами или чек-листами.

Визуальный формат нужен не для украшения, а для быстрого распознавания:

- главной идеи;
- простой формулы;
- типичной ошибки;
- обязательного правила;
- практического приёма;
- следующего шага.

### 16.1. Обязательные callout-блоки

Документ должен использовать callout-блоки Obsidian, если они помогают отделить разные виды информации.

Рекомендуемый набор:

```md
> [!info] Главное
> Краткая суть темы или этапа.
```

```md
> [!tip] Простая формула
> Проверяемая формула мышления или короткое правило применения.
```

```md
> [!warning] Не путать
> Уточнение, с чем нельзя смешивать понятие, этап или решение.
```

```md
> [!important] Правило
> Обязательное правило, которое влияет на проектное решение.
```

```md
> [!note] Практический приём
> Способ применения правила на практике.
```

Callout-блок запрещено использовать для эмоционального текста, декоративного вступления или повторения уже сказанного без нового смысла.

### 16.2. Размещение callout-блоков

Для энциклопедической статьи рекомендуется следующий порядок:

1. После назначения документа: `[!info] Главное`.
2. После определения: `[!tip] Простая формула`.
3. Перед классификацией или видами: `[!warning] Не путать`.
4. Перед правилами анализа: `[!important] Правило`.
5. Перед примерами: `[!note] Практический приём`.

Для roadmap-документа рекомендуется использовать:

- `[!info] Главное` после назначения;
- `[!warning] Не путать` в разделе границ этапа;
- `[!important] Правило` перед правилами этапа;
- `[!note] Практический приём` перед порядком работы или примерами.

Для анкеты рекомендуется использовать:

- `[!info] Главное` после назначения анкеты;
- `[!warning] Не путать` в правилах заполнения;
- `[!important] Критерий` перед критериями полноты ответов.

### 16.3. Правило таблиц

Таблица используется, если нужно сравнить:

- понятия;
- виды;
- уровни;
- статусы;
- критерии;
- входы и выходы;
- ошибки и реакции.

Таблица не должна заменять проверяемые правила и критерии завершения.

### 16.4. Правило истории изменений

Каждый крупный документ должен иметь раздел `История изменений`.

История изменений должна фиксировать смысловые изменения:

- создан документ;
- добавлены правила;
- обновлены ссылки;
- добавлены диаграммы;
- уточнены критерии завершения;
- изменён формат документа.

История изменений не должна фиксировать мелкие правки пунктуации, если они не меняют смысл документа.

## 17. Связанные документы

- [[PROJECT_SCOPE|PROJECT_SCOPE]]
  - Передаёт: масштаб проекта.
  - Используется для: проверки, что документ не выходит за границы базы знаний.
  - Ограничение: не задаёт стиль изложения.

- [[Digital_System_CAD_Philosophical_Essay_for_Codex|Digital System CAD Philosophical Essay]]
  - Передаёт: принципы structured facts, typed relations, definitions, views и provisional metamodel.
  - Используется для: ужесточения правил изложения документов Digital System CAD.
  - Ограничение: не заменяет регламент письма.

- [[docs/01_regulations/Documentation_System_Regulation|Documentation System Regulation]]
  - Передаёт: систему документации.
  - Используется для: выбора типа документа и места в слоях.
  - Ограничение: не заменяет правила письма.

- [[docs/01_regulations/Link_Rules|Link Rules]]
  - Передаёт: правила ссылок.
  - Используется для: оформления связанных документов.
  - Ограничение: не определяет качество формулировок.

- [[docs/01_regulations/Diagram_Rules|Diagram Rules]]
  - Передаёт: правила визуализации.
  - Используется для: выбора диаграмм.
  - Ограничение: не заменяет текстовое содержание.

## 18. Следующий шаг

После работы с регламентом необходимо применять его при создании, проверке и обновлении связанных документов.

# Регламент системы документации цифровых проектов

## 1. Назначение регламента

Регламент определяет правила построения системы документации проекта Programming Digital Systems.

Регламент применяется ко всем документам проекта: roadmap-документам, анкетам, энциклопедическим статьям, учебным главам, техническим требованиям, архитектурным описаниям, шаблонам и примерам.

> [!info] Главное
> Документ задаёт правило, которое должно применяться при создании и изменении материалов проекта.

## 2. Область применения

Регламент применяется при создании документов для следующих областей цифровых систем:

- скрипты автоматизации;
- GUI-приложения;
- web-системы;
- embedded-системы;
- PLC-системы;
- CNC/CAM-системы;
- базы данных;
- интеграционные системы.

## 3. Главный принцип

Документация должна быть не набором отдельных файлов, а связанной системой знаний.

Каждый документ должен показывать:

- откуда пришла информация;
- какое решение из неё следует;
- в какой следующий документ она передаётся;
- как пользователь должен двигаться дальше.

## 4. Типы документов

В проекте используются следующие типы документов:

### 4.1. Регламент

Определяет правила работы с документацией, структурой, ссылками, диаграммами и стилем изложения.

### 4.2. Roadmap-документ

Ведёт пользователя по проектному этапу и фиксирует порядок проектирования.

### 4.3. Анкета

Преобразует правила roadmap-документа в последовательность вопросов.

### 4.4. Энциклопедическая статья

Раскрывает отдельное понятие цифрового мира.

### 4.5. Пример

Показывает применение правила в конкретной области цифровых систем.

### 4.6. Диаграмма

Визуально отражает структуру, поток, состояние, последовательность или взаимосвязь.

## 5. Правило создания нового документа

Новый документ допускается создавать только если определены:

- назначение документа;
- тип документа;
- слой документации;
- место в маршруте разработки;
- входные документы;
- выходные документы;
- связанные понятия;
- необходимость диаграмм;
- критерии завершения;
- следующий шаг пользователя.

Документ запрещено создавать как изолированную заметку, если он должен быть частью roadmap, анкеты, энциклопедии, примеров, диаграмм или чек-листов.

Если документ создаётся временно как черновик, он должен быть помечен как черновик и не должен добавляться в карту документации как завершённый элемент.

## 6. Порядок создания документа

Перед созданием нового документа необходимо выполнить следующие шаги:

1. Определить проблему или проектный вопрос, который закрывает документ.
2. Проверить, не существует ли уже документ с таким назначением.
3. Выбрать слой документации.
4. Выбрать тип документа.
5. Определить входные и выходные документы.
6. Определить обязательные разделы по типу документа.
7. Определить, нужны ли диаграммы.
8. Определить, нужно ли обновить карту документации.
9. Создать документ.
10. Проверить ссылки, критерии завершения и следующий шаг.

Документ считается созданным корректно, если после его добавления пользователь понимает:

- зачем документ существует;
- когда его нужно читать;
- что уже должно быть известно до чтения;
- какой результат должен появиться после работы с документом;
- куда двигаться дальше.

## 7. Правило выбора типа документа

Тип документа выбирается по назначению.

| Назначение | Тип документа |
|---|---|
| Провести пользователя по проектному этапу | Roadmap-документ |
| Собрать проектные ответы | Анкета |
| Объяснить понятие | Энциклопедическая статья |
| Показать применение правила | Пример |
| Проверить готовность этапа | Чек-лист |
| Показать структуру или поток | Диаграмма |
| Зафиксировать правило документации | Регламент |

Если документ одновременно пытается быть roadmap, энциклопедией и примером, его необходимо разделить на несколько документов.

## 8. Минимальная структура любого документа

Любой крупный документ должен содержать:

- название;
- назначение;
- место в системе знаний или маршруте;
- область применения;
- основные правила, понятия или вопросы;
- связанные документы;
- критерии завершения или критерии готовности;
- историю изменений.

Если документ не имеет критериев завершения, он не должен считаться готовым.

Если документ связан с другими документами, раздел `Связанные документы` обязателен.

## 9. Правило обновления карт

После создания нового крупного документа необходимо проверить, нужно ли обновить:

- [[docs/00_maps/00_Documentation_Map|Documentation Map]];
- [[docs/00_maps/00_Knowledge_Layer_Map|Knowledge Layer Map]];
- [[docs/00_maps/00_Development_Route_Map|Development Route Map]];
- связанные roadmap-документы;
- связанные анкеты;
- связанные чек-листы;
- связанные диаграммы.

Карта документации должна обновляться, если новый документ:

- добавляет новый слой;
- добавляет новый важный маршрут;
- добавляет новый основной документ слоя;
- меняет порядок движения пользователя;
- становится входным или выходным документом для существующих этапов.

## 10. Правило жизненного статуса документа

Документ может иметь один из статусов:

- черновик;
- рабочая версия;
- готов к применению;
- требует пересмотра;
- устарел.

Статус `готов к применению` допускается только если:

- документ имеет назначение;
- связи оформлены по правилам ссылок;
- критерии завершения определены;
- открытые вопросы вынесены отдельно;
- документ не противоречит карте документации;
- документ не содержит битых ссылок.

## 11. Правило маршрута

Каждый roadmap-документ должен иметь:

- входные условия;
- порядок заполнения;
- контрольные вопросы;
- критерии завершения;
- выходные данные для следующего документа.

## 12. Правило связности

Каждый документ должен иметь раздел `Связанные документы`, если он влияет на другие документы или зависит от них.

Связь должна объяснять не только факт ссылки, но и смысл связи.

## 13. Правило визуальной информативности

Если информация описывает структуру, связь, поток, состояние, последовательность или иерархию, документ должен содержать визуальное представление.

Тип диаграммы должен соответствовать типу информации.

## 14. Правило анкет

Для каждого важного проектного этапа должна существовать связанная анкета.

Анкета должна обеспечивать движение от неопределённой идеи к зафиксированному проектному решению.

## 15. Правило масштабируемости

Документы должны быть пригодны для роста проекта до уровня энциклопедии, учебного курса, инженерного справочника и полной базы знаний.

Нельзя писать документы так, будто они существуют только для одной локальной задачи.

## 16. Связанные документы

- [[PROJECT_SCOPE|PROJECT_SCOPE]]
  - Передаёт: масштаб и назначение проекта.
  - Используется для: проверки, что правила создания документов соответствуют общей цели базы знаний.
  - Ограничение: не задаёт подробный порядок создания документов.

- [[docs/01_regulations/Document_Writing_Rules|Document Writing Rules]]
  - Передаёт: стиль и правила изложения.
  - Используется для: проверки качества текста нового документа.
  - Ограничение: не определяет карту слоёв.

- [[docs/01_regulations/Link_Rules|Link Rules]]
  - Передаёт: правила ссылок.
  - Используется для: оформления связей между документами.
  - Ограничение: не определяет содержание документа.

- [[docs/01_regulations/Diagram_Rules|Diagram Rules]]
  - Передаёт: правила диаграмм.
  - Используется для: решения, нужна ли визуальная схема.
  - Ограничение: не заменяет текст документа.

- [[docs/02_templates/Roadmap_Document_Template|Roadmap Document Template]]
  - Передаёт: структуру roadmap-документа.
  - Используется для: создания новых roadmap-документов.
  - Ограничение: не содержит содержание конкретного этапа.

- [[docs/02_templates/Questionnaire_Document_Template|Questionnaire Document Template]]
  - Передаёт: структуру анкеты.
  - Используется для: создания новых анкет.
  - Ограничение: не содержит конкретные вопросы этапа.

## 17. Следующий шаг

После работы с регламентом необходимо применять его при создании, проверке и обновлении связанных документов.

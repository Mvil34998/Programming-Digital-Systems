# Questionnaire: System Design / Анкета проектирования системы

## 1. Назначение анкеты

`01_Questionnaire_System_Design.md` превращает [[docs/03_roadmaps/01_Roadmap_System_Design|Roadmap: System Design]] в последовательность вопросов для совместной работы пользователя и чат-бота над проектированием цифровой системы.

Анкета должна быть заполнена так, чтобы чат-бот мог читать `.md` файл, понимать проектные ответы без лишних догадок и задавать уточняющие вопросы только там, где информация явно отсутствует, противоречива или помечена как неизвестная.

Анкета должна помочь определить:

- задачу системы;
- границу системы;
- внешних участников;
- сущности;
- данные;
- правила;
- состояния;
- события;
- потоки;
- хранение;
- ошибки;
- открытые вопросы.

Анкета не должна выбирать инструменты реализации и не должна проектировать архитектуру системы.

> [!info] Главное
> Анкета является рабочим интерфейсом между пользователем и чат-ботом. Ответы должны быть достаточно структурированы, чтобы бот мог продолжить проектирование без домыслов.

## 2. Связанный roadmap-документ

- [[docs/03_roadmaps/01_Roadmap_System_Design|Roadmap: System Design]]
  - Передаёт: порядок проектирования системы.
  - Используется для: построения вопросов анкеты.
  - Ограничение: анкета фиксирует ответы пользователя и не заменяет roadmap-документ.

## 3. Правила заполнения для пользователя

- Каждый ответ должен быть конкретным.
- Если ответ неизвестен, нужно писать `UNKNOWN`, а не оставлять поле пустым.
- Если вопрос не относится к проекту, нужно писать `NOT_APPLICABLE` и кратко объяснять причину.
- Если ответ частичный, нужно писать `PARTIAL` и перечислять, чего не хватает.
- Нельзя писать `потом решим`, если вопрос влияет на следующий этап.
- Нельзя выбирать библиотеки, базы данных, GUI-фреймворки и другие инструменты реализации.
- Нельзя смешивать категории и примеры на одном уровне.
- Открытые вопросы должны фиксироваться отдельно и иметь владельца ответа или источник уточнения.

## 4. Правила чтения анкеты для чат-бота

Чат-бот должен читать анкету как источник проектных фактов.

Чат-боту запрещено:

- додумывать ответ, если поле пустое или содержит `UNKNOWN`;
- превращать предположение в факт;
- выбирать инструмент реализации на этапе проектирования системы;
- игнорировать открытые вопросы;
- менять смысл ответа пользователя без явной пометки.

Чат-бот должен:

- использовать только зафиксированные ответы как факты;
- отличать `ANSWERED`, `PARTIAL`, `UNKNOWN` и `NOT_APPLICABLE`;
- задавать уточняющие вопросы по полям `UNKNOWN` и `PARTIAL`;
- сохранять идентификаторы вопросов;
- переносить блокирующие неопределённости в раздел `Открытые вопросы`;
- формировать следующий проектный шаг только на основе заполненных данных.

> [!warning] Не путать
> Анкета не должна заставлять чат-бота угадывать. Если данных не хватает, это не ошибка пользователя, а явный открытый вопрос.

## 5. Формат ответа

Каждый вопрос содержит блок `Ответ` в формате YAML.

Допустимые значения `answer_status`:

- `TODO` — ответ ещё не заполнен;
- `ANSWERED` — ответ заполнен достаточно;
- `PARTIAL` — ответ заполнен частично;
- `UNKNOWN` — пользователь не знает ответ;
- `NOT_APPLICABLE` — вопрос не относится к проекту.

Базовый формат:

```yaml
answer_status: TODO
user_answer: ""
facts: []
constraints: []
unknowns: []
not_assume: []
open_questions: []
bot_next_action: ""
```

Смысл полей:

| Поле | Назначение |
|---|---|
| `answer_status` | Статус заполнения ответа |
| `user_answer` | Краткий ответ пользователя человеческим языком |
| `facts` | Факты, которые бот может использовать без уточнения |
| `constraints` | Ограничения, которые нельзя нарушать |
| `unknowns` | Что пока неизвестно |
| `not_assume` | Что боту запрещено додумывать |
| `open_questions` | Вопросы, которые нужно уточнить |
| `bot_next_action` | Что бот должен сделать после чтения ответа |

## 6. Входные данные для анкеты

Перед заполнением анкеты должны быть известны хотя бы частично:

- идея системы;
- назначение системы;
- основная проблема;
- ожидаемый результат;
- основной пользователь или внешний потребитель результата;
- известные входные данные, файлы, команды, сигналы или источники.

Если эти данные неизвестны, первый блок анкеты заполняется со статусом `PARTIAL` или `UNKNOWN`.

## 7. Вопросы анкеты

## Q-SD-001. Какую задачу должна решать система?

### Вопрос

Какую конкретную задачу должна решать система?

### Пояснение

Ответ должен описывать не инструмент и не способ реализации, а результат, который система должна обеспечить.

### Ответ

```yaml
answer_status: TODO
user_answer: ""
facts:
  problem: ""
  work_object: ""
  expected_result: ""
  user_or_consumer: ""
constraints: []
unknowns: []
not_assume:
  - "Не выбирать инструменты реализации."
open_questions: []
bot_next_action: "Если все поля facts заполнены, перейти к Q-SD-002. Если нет, задать уточняющий вопрос о задаче системы."
```

### Критерий заполнения

Ответ считается заполненным, если указаны проблема, объект работы системы, ожидаемый результат и пользователь или внешний потребитель результата.

## Q-SD-002. Где проходит граница системы?

### Вопрос

Что входит в систему и что находится вне системы?

### Пояснение

Необходимо отделить ответственность системы от пользователя, внешних программ, оборудования и ручных действий.

### Ответ

```yaml
answer_status: TODO

  inside_system: []
  outside_system: []
  external_actors: []
  external_data_sources: []
  external_result_consumers: []
constraints: []
unknowns: []

open_questions: []
bot_next_action: "Если граница системы ясна, перейти к Q-SD-003. Если граница размыта, спросить, кто или что отвечает за спорные действия."
```

### Критерий заполнения

Ответ считается заполненным, если указаны элементы внутри системы, элементы вне системы, внешние участники, внешние источники данных и внешние потребители результата.

## Q-SD-003. Какие сущности существуют в системе?

### Вопрос

Какие значимые объекты предметной области, данных, интерфейса, системы и результата должна учитывать система?

### Пояснение

Сущность должна иметь смысловую роль. Нельзя включать слово в список сущностей только потому, что оно встречается в описании проекта.

### Ответ

```yaml
answer_status: TODO

facts:
  domain_entities: []
  information_entities: []
  system_entities: []
  interface_entities: []
  result_entities: []
relationships: []
unknowns: []
not_assume:
  - "Не превращать каждую переменную или файл в сущность без смысловой роли."
open_questions: []
bot_next_action: "Если сущности названы и связаны, перейти к Q-SD-004. Если связи неясны, запросить связи между сущностями."
```

### Критерий заполнения

Ответ считается заполненным, если для каждой важной сущности указаны название, назначение, тип сущности, связь с другими сущностями и открытые вопросы, если они есть.

## 8. Контроль полноты ответов

> [!important] Критерий
> Анкета готова для следующего этапа только если чат-бот может выделить факты, ограничения, неизвестные вопросы и следующий шаг без самостоятельных догадок.

Анкета считается заполненной достаточно для перехода к [[docs/03_roadmaps/02_Roadmap_System_Architecture_Design|Roadmap: System Architecture Design]], если:

- определена граница системы;
- определены внешние участники;
- определены сущности;
- определены данные;
- определены правила;
- определены состояния;
- определены события;
- определены потоки;
- определено хранение;
- определены ошибки;
- открытые вопросы вынесены отдельно;
- ответы не содержат выбора инструментов реализации;
- каждое неизвестное поле помечено как `UNKNOWN`, `PARTIAL` или `NOT_APPLICABLE`;
- чат-боту явно указано, что нельзя додумывать.

## 9. Выходные данные анкеты

После заполнения анкеты должны быть получены:

- описание задачи системы;
- граница системы;
- список внешних участников;
- список сущностей;
- список данных;
- список правил;
- список состояний;
- список событий;
- список потоков;
- список хранимых данных;
- список ошибок;
- список открытых вопросов;
- входные данные для [[docs/04_questionnaires/02_Questionnaire_System_Architecture_Design|Questionnaire: System Architecture Design]].

## 10. Открытые вопросы

```yaml
open_questions:
  - id: Q-OPEN-001
    question: ""
    why_important: ""
    blocks_stage: ""
    answer_owner: ""
    current_status: TODO
```

Открытый вопрос должен содержать:

- идентификатор;
- формулировку вопроса;
- почему он важен;
- какой этап он блокирует;
- кто или что может дать ответ;
- текущий статус.

## 11. Следующий шаг

После заполнения анкеты необходимо передать ответы в следующий roadmap-этап или проверить готовность по связанному чек-листу.

## 12. Связанные документы

### Входные документы

- [[docs/03_roadmaps/01_Roadmap_System_Design|Roadmap: System Design]]
  - Передаёт: порядок проектирования системы.
  - Используется для: структуры вопросов анкеты.
  - Ограничение: не хранит ответы конкретного проекта.

- [[docs/02_templates/Questionnaire_Document_Template|Questionnaire Document Template]]
  - Передаёт: общий формат анкеты.
  - Используется для: согласования структуры вопросов.
  - Ограничение: не содержит специальные правила совместной работы с чат-ботом.

### Выходные документы

- [[docs/04_questionnaires/01_Questionnaire_System_Design_Filled_Python_File_Processing_Utility|Filled Questionnaire: Python File Processing Utility]]
  - Получает: структуру анкеты как тестовый пример заполнения.
  - Используется для: проверки, понимает ли чат-бот заполненную анкету без домыслов.
  - Ограничение: является примером, а не универсальной анкетой.

- [[docs/03_roadmaps/02_Roadmap_System_Architecture_Design|Roadmap: System Architecture Design]]
  - Получает: проектные ответы для перехода к архитектуре системы.
  - Используется для: проектирования слоёв, модулей, моделей, интерфейсов и зависимостей.
  - Ограничение: не должен менять ответы анкеты без явного уточнения.

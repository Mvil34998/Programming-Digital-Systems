# 06. Checklist: Implementation Architecture / Архитектура реализации

## 1. Назначение документа

`06_Checklist_Implementation_Architecture.md` проверяет готовность структуры реализации перед переходом к написанию кода.

Чек-лист подтверждает, что проект имеет понятную структуру файлов, модулей, точек входа, зависимостей, конфигурации, логирования и тестов.

> [!info] Главное
> Чек-лист проверяет готовность результата перед переходом к следующему этапу.

## 3. Чек-лист готовности

### 3.1. Структура проекта

- [ ] Определено дерево папок проекта.
- [ ] Определено назначение каждой основной папки.
- [ ] Определены точки входа в приложение или утилиту.
- [ ] Определено, где хранятся конфигурации.
- [ ] Определено, где хранятся тесты.
- [ ] Определено, где хранятся документация и примеры.

### 3.2. Модули реализации

- [ ] Определены основные модули реализации.
- [ ] Каждый модуль связан с архитектурным компонентом.
- [ ] Для каждого модуля указана ответственность.
- [ ] Для каждого модуля указаны входы и выходы.
- [ ] Модули не содержат смешанную ответственность.

### 3.3. Данные и модели

- [ ] Определены структуры данных реализации.
- [ ] Определено, какие модели используются внутри системы.
- [ ] Определено, какие DTO, схемы или форматы используются на границах системы.
- [ ] Определено, где выполняется валидация данных.
- [ ] Определено, где выполняется преобразование данных.

### 3.4. Адаптеры и внешние интерфейсы

- [ ] Определены файловые адаптеры.
- [ ] Определены сетевые или API-адаптеры, если они есть.
- [ ] Определены аппаратные адаптеры, если они есть.
- [ ] Определены промышленные адаптеры, если они есть.
- [ ] Внешние интерфейсы отделены от основной логики.

### 3.5. Конфигурация и окружение

- [ ] Определены конфигурационные параметры.
- [ ] Определено, какие параметры обязательны.
- [ ] Определено, где конфигурация хранится физически.
- [ ] Определено, как проверяется корректность конфигурации.
- [ ] Определено, как система сообщает об ошибке конфигурации.

### 3.6. Ошибки и логирование

- [ ] Определены типы ошибок реализации.
- [ ] Определено, какие ошибки логируются.
- [ ] Определено, где сохраняются логи.
- [ ] Определено, какие сообщения видит пользователь или оператор.
- [ ] Определено, какие ошибки блокируют выполнение.

### 3.7. Тестовая структура

- [ ] Определена структура тестов.
- [ ] Определены тестовые данные.
- [ ] Определено, какие модули покрываются unit-тестами.
- [ ] Определено, какие сценарии покрываются integration-тестами.
- [ ] Определено, как запускать тесты.

## 4. Блокирующие условия

Переход к коду запрещён, если:

- структура проекта не определена;
- модули не имеют ответственности;
- внешние интерфейсы смешаны с основной логикой;
- конфигурация не определена;
- ошибки и логирование не спроектированы;
- тестовая структура отсутствует.

## 5. Результат проверки

- [ ] Этап готов к реализации кода.
- [ ] Этап требует доработки.
- [ ] Этап заблокирован открытыми вопросами.

## 6. Следующий шаг

После проверки чек-листа необходимо либо перейти к следующему этапу, либо вернуть невыполненные пункты в открытые вопросы.

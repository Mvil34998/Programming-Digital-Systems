# 01. Checklist: System Design / Проектирование системы

## 1. Назначение документа

`01_Checklist_System_Design.md` проверяет готовность этапа проектирования системы перед переходом к проектированию архитектуры системы.

Чек-лист не собирает новые ответы. Он проверяет, достаточно ли качественно заполнены результаты этапа.

> [!info] Главное
> Чек-лист проверяет готовность результата перед переходом к следующему этапу.

## 2. Проверяемые документы

- [[docs/03_roadmaps/01_Roadmap_System_Design|Roadmap: System Design]]
  - Передаёт: порядок проектирования системы.
  - Используется для: проверки обязательных результатов этапа.

- [[docs/04_questionnaires/01_Questionnaire_System_Design|Questionnaire: System Design]]
  - Передаёт: ответы по проектированию системы.
  - Используется для: проверки заполненности решений.

  - Передаёт: визуальные модели этапа.
  - Используется для: проверки связей между элементами системы.

## 3. Чек-лист готовности

### 3.1. Граница системы

- [ ] Определено, что входит в систему.
- [ ] Определено, что находится вне системы.
- [ ] Определены внешние участники.
- [ ] Определены внешние источники данных, команд, сигналов или файлов.
- [ ] Определены внешние потребители результата.

### 3.2. Сущности

- [ ] Определены предметные сущности.
- [ ] Определены информационные сущности.
- [ ] Определены системные сущности.
- [ ] Определены интерфейсные сущности, если система имеет интерфейс.
- [ ] Определены результирующие сущности.
- [ ] Для каждой сущности указано назначение.
- [ ] Для каждой сущности указаны связи с другими элементами системы.

### 3.3. Данные

- [ ] Определены входные данные.
- [ ] Определены внутренние данные.
- [ ] Определены хранимые данные.
- [ ] Определены выходные данные.
- [ ] Определены конфигурационные данные.
- [ ] Для каждого вида данных указан источник.
- [ ] Для каждого вида данных указано назначение.
- [ ] Для каждого вида данных указаны правила проверки.

### 3.4. Правила

- [ ] Определены правила проверки данных.
- [ ] Определены правила обработки.
- [ ] Определены правила расчётов, если расчёты есть.
- [ ] Определены правила переходов состояний.
- [ ] Определены правила реакции на события.
- [ ] Определены правила ошибок.
- [ ] Для каждого правила указано условие применения.
- [ ] Для каждого правила указан результат применения.

### 3.5. Состояния

- [ ] Определены состояния системы.
- [ ] Определены состояния ключевых сущностей.
- [ ] Определены состояния процессов.
- [ ] Определены состояния данных.
- [ ] Определены ошибочные состояния.
- [ ] Для каждого состояния указаны условия входа.
- [ ] Для каждого состояния указаны условия выхода.
- [ ] Для каждого состояния указаны допустимые и запрещённые действия.

### 3.6. События

- [ ] Определены пользовательские события.
- [ ] Определены системные события.
- [ ] Определены события данных.
- [ ] Определены события времени, если они есть.
- [ ] Определены события оборудования или внешних систем, если они есть.
- [ ] Для каждого события указан источник.
- [ ] Для каждого события указана реакция системы.

### 3.7. Потоки

- [ ] Определены потоки данных.
- [ ] Определены потоки управления.
- [ ] Определены пользовательские потоки, если пользователь участвует.
- [ ] Определены потоки событий.
- [ ] Определены потоки ошибок.
- [ ] Для каждого потока указан источник.
- [ ] Для каждого потока указан получатель.
- [ ] Для каждого потока указан передаваемый объект.

### 3.8. Хранение

- [ ] Определено, какие данные должны сохраняться.
- [ ] Определено, зачем эти данные сохраняются.
- [ ] Определён срок жизни хранимых данных.
- [ ] Определены правила обновления данных.
- [ ] Определены требования к целостности данных.
- [ ] Выбор конкретной базы данных или формата хранения не выполнен преждевременно.

### 3.9. Ошибки

- [ ] Определены ошибки входных данных.
- [ ] Определены ошибки обработки.
- [ ] Определены ошибки состояния.
- [ ] Определены ошибки хранения.
- [ ] Определены ошибки интеграции, если интеграция есть.
- [ ] Для каждой ошибки указан источник.
- [ ] Для каждой ошибки указана причина.
- [ ] Для каждой ошибки указана реакция системы.
- [ ] Для каждой ошибки указан уровень критичности.

## 4. Блокирующие условия

Переход к архитектуре системы запрещён, если:

- граница системы не определена;
- сущности не связаны с данными, правилами, состояниями или потоками;
- неизвестно, какие данные входят и какие выходят;
- правила поведения заменены общими пожеланиями;
- ошибки не определены;
- открытые вопросы скрыты внутри текста и не вынесены отдельно.

## 5. Результат проверки

- [ ] Этап готов к переходу в [[docs/03_roadmaps/02_Roadmap_System_Architecture_Design|Roadmap: System Architecture Design]].
- [ ] Этап требует доработки.
- [ ] Этап заблокирован открытыми вопросами.

## 6. Следующий шаг

После проверки чек-листа необходимо либо перейти к следующему этапу, либо вернуть невыполненные пункты в открытые вопросы.

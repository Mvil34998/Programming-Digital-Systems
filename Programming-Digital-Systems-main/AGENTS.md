# AGENTS.md

## 1. Назначение файла

Этот файл определяет правила работы AI-агента с репозиторием `Programming-Digital-Systems`.

AI-агент должен использовать этот файл перед созданием, изменением или анализом документов проекта.

Главная задача агента — сохранять масштаб, связность, структуру и стиль базы знаний по проектированию и разработке цифровых систем, а также удерживать конечную цель проекта: проверку гипотезы [[Digital_System_CAD_Concept_for_Codex|Digital System CAD]].

## 2. Масштаб проекта

Проект не является набором отдельных заметок.

Проект развивается как универсальная связанная база знаний по проектированию, разработке, документированию, проверке, сопровождению и развитию цифровых систем.

Долгосрочная цель проекта — выяснить, можно ли описывать большинство цифровых систем через связанную инженерную метамодель и использовать её как основу для SDD, анкет, таблиц, диаграмм, задач реализации, тестов и контекста для код-бота.

Текущий этап не является разработкой приложения или metamodeling workbench. Текущий этап — сбор, анализ и формирование информации о методах метамоделирования и цифровых системах.

Проект должен быть пригоден для следующих форматов:

- учебный курс;
- инженерный справочник;
- система roadmap-документов;
- система проектных анкет;
- энциклопедия понятий;
- Obsidian-база;
- GitHub-документация;
- методология проектирования цифровых систем;
- исследовательская база для Digital System CAD.

## 3. Обязательные документы для чтения перед работой

Перед созданием или редактированием документов AI-агент должен прочитать следующие документы:

1. `PROJECT_SCOPE.md`
   - Определяет масштаб проекта, центральную формулу цифровой системы и области применения.

2. `Digital_System_CAD_Concept_for_Codex.md`
   - Определяет конечную цель проекта, гипотезу метамодели, роль SDD и правила работы Codex.

3. `Digital_System_CAD_Philosophical_Essay_for_Codex.md`
   - Определяет философские основания проекта: typed elements, typed relations, definitions, views, constraints, traceability и различение model / metamodel / implementation.

4. `docs/00_maps/00_Documentation_Map.md`
   - Определяет карту базы знаний, слои документации и маршрут движения пользователя.

5. `docs/01_regulations/Documentation_System_Regulation.md`
   - Определяет общие правила построения системы документации.

6. `docs/01_regulations/Document_Writing_Rules.md`
   - Определяет стиль изложения, запреты, проверяемость и правила отделения категорий от примеров.

7. `docs/01_regulations/Link_Rules.md`
   - Определяет правила ссылок между документами, разделами, смысловыми блоками, цитатами и диаграммами.

8. `docs/01_regulations/Diagram_Rules.md`
   - Определяет правила выбора и оформления диаграмм.

9. `docs/02_templates/Roadmap_Document_Template.md`
   - Используется перед созданием roadmap-документов.

10. `docs/02_templates/Questionnaire_Document_Template.md`
   - Используется перед созданием анкет.

## 5. Главные правила создания документов

AI-агент должен соблюдать следующие правила:

- документ должен быть частью общей системы знаний;
- документ должен иметь назначение;
- документ должен показывать своё место в маршруте разработки;
- документ должен иметь связи с другими документами, если такие связи существуют;
- документ должен различать входные и выходные документы;
- документ должен содержать проверяемые формулировки;
- документ не должен содержать личный шум;
- документ не должен смешивать требования, инструменты и реализацию;
- документ не должен смешивать категории и примеры на одном уровне;
- документ должен использовать диаграммы, если информация требует визуального представления;
- документ должен быть пригоден для расширения до уровня энциклопедии, учебного курса или инженерного справочника;
- документ должен показывать, если это применимо, какой элемент будущей модели Digital System CAD он уточняет: требование, сущность, данные, правило, состояние, событие, поток, хранилище, интерфейс, ошибка, тест, задача или кодовый артефакт.

## 6. Запрещено

AI-агенту запрещено:

- создавать изолированные документы без места в общей системе;
- создавать документы без раздела назначения;
- создавать roadmap-документы без входных условий и критериев завершения;
- создавать анкеты без критериев заполнения вопросов;
- добавлять случайные ссылки без объяснения смысла связи;
- использовать декоративный текст;
- использовать эмоциональные оценки;
- писать свободные рассуждения вместо правил;
- смешивать виды документов с примерами этих видов;
- выбирать инструменты реализации внутри технических требований, если это не внешний обязательный constraint;
- добавлять диаграммы только ради оформления;
- развивать документы так, будто целью является только учебный текст, без связи с метамоделью, SDD и проверкой применимости Digital System CAD.

## 7. Правило Digital System CAD

Перед содержательным расширением документа агент должен определить, как правка связана с конечной целью проекта:

- уточняет ли она базовый элемент цифровой системы;
- уточняет ли она связь между элементами;
- помогает ли она собрать SDD из модели;
- помогает ли она превратить анкету в структурированный сбор данных;
- помогает ли она проверить модель на практическом примере;
- помогает ли она отделить универсальное ядро от доменного расширения.

Если правка не влияет на понимание, моделирование, проверку, реализацию или сопровождение цифровой системы, её не нужно добавлять на раннем этапе.

Codex не должен называть текущую работу разработкой metamodeling workbench. Workbench является возможным будущим направлением реализации, если рабочая форма метамодели окажется полезной.

Модель Digital System CAD должна описывать не только элементы, но и структурированные факты:

```text
Element + typed relation + meaning + constraint + traceability
```

Codex должен соблюдать следующие уточнения:

- отношение является объектом модели первого класса, а не декоративной линией диаграммы;
- имя элемента не заменяет определение;
- важный элемент должен иметь Definition, Purpose, Context и Source;
- diagram, table, questionnaire, SDD и Codex context являются views или outputs, а не отдельными источниками правды;
- если смысл элемента или связи неясен, нужно фиксировать Open question, а не молча придумывать смысл;
- Codex должен различать real system, model, metamodel, meta-metamodel, view, transformation и implementation.

## 8. Правило ссылок

Если документ связан с другим документом, ссылка должна объяснять смысл связи.

Неправильно:

```md
- `01_Roadmap_System_Design.md`
```

Правильно:

```md
- `01_Roadmap_System_Design.md`
  - Передаёт: сущности, данные, правила, состояния, события, потоки, хранение и ошибки.
  - Используется для: формирования технических требований.
  - Ограничение: не должен содержать выбор конкретных библиотек и фреймворков.
```

## 9. Правило диаграмм

Если информация описывает структуру, связь, поток, состояние, последовательность или иерархию, агент должен рассмотреть необходимость диаграммы.

Тип диаграммы должен соответствовать смыслу информации:

- маршрут или алгоритм — `flowchart`;
- сущности и связи — `classDiagram`;
- состояния — `stateDiagram`;
- последовательность взаимодействий — `sequenceDiagram`;
- иерархия понятий — `mindmap` или `flowchart`;
- архитектурные уровни — C4 или совместимая flowchart-адаптация.

Если документ предназначен для Obsidian, Mermaid-синтаксис должен быть максимально совместимым.

## 10. Правило анкет

Анкета должна вести пользователя от неопределённой идеи к проектному решению.

Каждый вопрос анкеты должен иметь:

- идентификатор;
- формулировку вопроса;
- пояснение;
- поле для ответа;
- критерий заполнения;
- связанные документы;
- следующий шаг.

В контексте Digital System CAD анкета должна стремиться к структурированному результату: ответ должен быть пригоден для создания или обновления элемента модели, а не только для чтения человеком.

## 11. Правило примеров

Если правило универсальное, примеры должны показывать применение в разных областях цифровых систем.

Категории примеров:

- скрипты автоматизации;
- GUI-приложения;
- web-системы;
- embedded-системы;
- PLC-системы;
- CNC/CAM-системы;
- базы данных;
- интеграционные системы.

Примеры должны быть вложены внутрь категорий, а не стоять на одном уровне с категориями.

## 12. Правило изменения структуры

AI-агент может создавать папки и подпапки для упорядочивания документов в Obsidian, если новая структура помогает навигации, разделяет разные типы документов или поддерживает будущую модель Digital System CAD.

Создание папки допускается только если агент определил:

- назначение папки;
- какие документы она будет содержать;
- к какому слою документации она относится;
- какие документы являются входными и выходными для этой группы;
- как папка будет отображаться в Obsidian через wikilinks и карты.

Папки не должны создаваться ради визуальной аккуратности без содержательной причины. Если достаточно существующего слоя или подпапки, нужно использовать существующую структуру.

Если агент создаёт новую папку, крупный документ, слой или маршрут, он должен проверить необходимость обновления:

- `docs/00_maps/00_Documentation_Map.md`;
- `docs/00_maps/00_Knowledge_Layer_Map.md`;
- связанных roadmap-документов;
- связанных анкет;
- связанных регламентов;
- связанных диаграмм.

## 13. Правило результата

Результат работы должен быть пригоден для практического использования.

Документ считается плохим, если он выглядит написанным, но не помогает пользователю двигаться дальше по проектному маршруту.

Документ считается хорошим, если он:

- объясняет место этапа;
- показывает вход и выход;
- содержит проверяемые правила;
- связан с другими документами;
- содержит нужные визуальные элементы;
- помогает пользователю принять следующее проектное решение.

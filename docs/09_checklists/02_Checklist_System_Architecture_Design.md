## 3. Чек-лист готовности

### 3.1. Архитектурная граница

- [ ] Подтверждена граница системы из этапа проектирования системы.
- [ ] Определено, какие внешние системы, пользователи, устройства или файлы взаимодействуют с системой.
- [ ] Определено, какие части являются внутренними, а какие внешними.

### 3.2. Архитектурные слои

- [ ] Определены основные слои системы.
- [ ] Для каждого слоя указано назначение.
- [ ] Для каждого слоя указано, какие данные он принимает и отдаёт.
- [ ] Запрещённые зависимости между слоями явно указаны.
- [ ] Слои не подменены конкретными библиотеками или фреймворками.

### 3.3. Модули и компоненты

- [ ] Определены основные модули системы.
- [ ] Для каждого модуля указана зона ответственности.
- [ ] Для каждого модуля указаны входы и выходы.
- [ ] Для каждого модуля указаны зависимости.
- [ ] Модули не дублируют ответственность друг друга.

### 3.4. Модели

- [ ] Определены ключевые модели данных или предметные модели.
- [ ] Модели связаны с сущностями из проектирования системы.
- [ ] Модели не смешаны с GUI, хранением или внешними форматами без необходимости.
- [ ] Для моделей указаны правила изменения и валидации.

### 3.5. Интерфейсы

- [ ] Определены пользовательские интерфейсы, если они есть.
- [ ] Определены программные интерфейсы, если они есть.
- [ ] Определены файловые интерфейсы, если они есть.
- [ ] Определены аппаратные или промышленные интерфейсы, если они есть.
- [ ] Для каждого интерфейса указаны входные и выходные данные.

### 3.6. Зависимости

- [ ] Определены внутренние зависимости между модулями.
- [ ] Определены внешние зависимости системы.
- [ ] Определено, какие зависимости критичны.
- [ ] Определено, какие зависимости можно заменить.
- [ ] Не выбран инструмент без связи с требованием или архитектурным ограничением.

### 3.7. Конфигурация

- [ ] Определено, какие параметры должны быть конфигурируемыми.
- [ ] Определено, кто изменяет конфигурацию.
- [ ] Определено, где конфигурация хранится логически.
- [ ] Определено, как система реагирует на ошибочную конфигурацию.

### 3.8. Точки расширения

- [ ] Определены места, где система может развиваться.
- [ ] Определены сценарии добавления новых форматов, модулей, интерфейсов или правил.
- [ ] Расширение не требует переписывания всей системы.
- [ ] Точки расширения не являются преждевременным усложнением.

## 4. Блокирующие условия

Переход к техническим требованиям запрещён, если:

- архитектурные слои не определены;
- ответственность модулей не разделена;
- интерфейсы не имеют входов и выходов;
- зависимости не зафиксированы;
- архитектура подменена выбором инструментов;
- неясно, какие решения должны стать техническими требованиями.

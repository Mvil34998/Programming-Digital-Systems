# 07. Checklist: Testing / Тестирование

## 1. Назначение документа

`07_Checklist_Testing.md` проверяет готовность тестирования перед переходом к эксплуатации.

Чек-лист подтверждает, что требования, модули, интерфейсы, ошибки и пользовательские сценарии имеют проверку.

## 3. Чек-лист готовности

### 3.1. Связь тестов с требованиями

- [ ] Для каждого обязательного требования есть проверка.
- [ ] Для каждого критичного требования есть критерий успешности.
- [ ] Невозможные или дорогие проверки вынесены в риски.
- [ ] Проверки не подменены визуальным осмотром без критерия.

### 3.2. Уровни тестирования

- [ ] Определены unit-тесты для изолированной логики.
- [ ] Определены integration-тесты для связей между модулями.
- [ ] Определены end-to-end сценарии, если система имеет полный пользовательский поток.
- [ ] Определены ручные проверки, если автоматизация проверки невозможна или нецелесообразна.
- [ ] Определены регрессионные проверки для критичных сценариев.

### 3.3. Тестовые данные

- [ ] Определены корректные тестовые данные.
- [ ] Определены ошибочные тестовые данные.
- [ ] Определены граничные значения.
- [ ] Определены пустые, неполные или повреждённые данные.
- [ ] Тестовые данные не смешаны с рабочими данными.

### 3.4. Ошибки и исключения

- [ ] Проверяются ошибки входных данных.
- [ ] Проверяются ошибки конфигурации.
- [ ] Проверяются ошибки хранения.
- [ ] Проверяются ошибки внешних систем или устройств, если они есть.
- [ ] Проверяются сообщения пользователю или оператору.
- [ ] Проверяется логирование ошибок.

### 3.5. Интерфейсы

- [ ] Проверяются пользовательские интерфейсы, если они есть.
- [ ] Проверяются файловые интерфейсы, если они есть.
- [ ] Проверяются API или сетевые интерфейсы, если они есть.
- [ ] Проверяются аппаратные или промышленные интерфейсы, если они есть.
- [ ] Проверяются форматы входа и выхода.

### 3.6. Запуск тестов

- [ ] Определено, как запускать тесты.
- [ ] Определено, какие тесты обязательны перед релизом.
- [ ] Определено, где хранится результат тестирования.
- [ ] Определено, кто анализирует результат тестирования.
- [ ] Определено, что делать при падении тестов.

## 4. Блокирующие условия

Переход к эксплуатации запрещён, если:

- обязательные требования не имеют проверок;
- критичные ошибки не проверяются;
- тестовые данные не определены;
- нет понятного способа запуска тестов;
- результат тестирования невозможно повторить.

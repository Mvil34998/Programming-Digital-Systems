# Development Route Diagrams / Диаграммы маршрута разработки

## 1. Назначение документа

`Development_Route_Diagrams.md` хранит крупные диаграммы маршрута разработки цифровой системы.

Документ показывает путь от идеи до проектирования, требований, выбора инструментария, реализации, тестирования, эксплуатации, сопровождения и развития.

Документ не заменяет [[docs/00_maps/Development_Route_Map|Development Route Map]]. Он визуализирует маршрут и его ключевые возвраты.

## 2. Связанные документы

### Входные документы

  - Ограничение: не заменяет roadmap выбора инструментария.

## 3. DG-ROUTE-001. Полный маршрут разработки

Назначение диаграммы: показать главный путь от идеи до развития системы.

```mermaid
flowchart TD
    A[Идея системы] --> B[Предметная область]
    B --> C[Проектирование системы]
    C --> D[Проектирование архитектуры системы]
    D --> E[Технические требования]
    E --> F[Связь требований с критериями выбора]
    F --> G[Выбор инструментария]
    G --> H[Архитектура реализации]
    H --> I[Код]
    I --> J[Тестирование]
    J --> K[Эксплуатация]
    K --> L[Сопровождение]
    L --> M[Развитие системы]
    M --> C
```

## 5. DG-ROUTE-003. Разделение требований и инструментария

```mermaid

    A --> D[Требования к хранению]
    A --> E[Требования к интерфейсам]
    A --> F[Требования к производительности]
    A --> G[Требования к ошибкам]

    B --> I[Критерии выбора]
    C --> I
    D --> I
    E --> I
    F --> I
    G --> I
    H --> I

    I --> J[Выбор инструментария]
    J --> K[Базовый инструментарий]
    J --> L[Прикладной инструментарий]
    J --> M[Специализированный инструментарий]
```

## 9. DG-ROUTE-007. Возвраты по маршруту

Назначение диаграммы: показать, что проблема или запрос может возвращать проект не только к коду, но и к требованиям, архитектуре или выбору инструментария.

```mermaid
flowchart TD
    A[Проблема или запрос] --> B{Тип изменения}
    B -->|Ошибка проектирования системы| C[Проектирование системы]
    B -->|Ошибка архитектуры системы| D[Проектирование архитектуры системы]
    B -->|Ошибка требования| E[Технические требования]
    B -->|Ошибка инструмента| F[Выбор инструментария]
    B -->|Ошибка реализации| G[Архитектура реализации или код]
    B -->|Ошибка теста| H[Тестирование]
    B -->|Ошибка эксплуатации| I[Эксплуатация]
    B -->|Дефект после выпуска| J[Сопровождение]
    B -->|Новая функция| K[Развитие системы]
```

## 10. DG-ROUTE-008. Запрещённые переходы

Назначение диаграммы: показать переходы, которые нарушают порядок разработки.

```mermaid
flowchart TD
    A[Идея] --> B[Код без проектирования]
    C[Проектирование системы] --> D[Выбор библиотеки без требований]
    E[Технические требования] --> F[Код без выбора инструментария]
    G[Выбор инструментария] --> H[Код без архитектуры реализации]
    I[Код] --> J[Эксплуатация без тестирования]
    K[Эксплуатация] --> L[Хаотичное исправление без сопровождения]
    M[Сопровождение] --> N[Новая функция без развития системы]
```

Все переходы на этой диаграмме являются запрещёнными.

## 12. Правила использования диаграмм маршрута

- Диаграммы маршрута используются для ориентации в процессе разработки.
- Диаграммы маршрута не заменяют roadmap-документы.
- Если маршрут меняется, нужно обновить [[docs/00_maps/Development_Route_Map|Development Route Map]] и этот документ.
- Если создаётся новая анкета или roadmap, нужно проверить, нужен ли новый узел на диаграммах маршрута.
- Запрещённые переходы должны оставаться явно видимыми.

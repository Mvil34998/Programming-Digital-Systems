# Examples Index / Индекс примеров

## 1. Назначение документа

`Examples_Index.md` определяет структуру слоя примеров проекта Programming Digital Systems.

Слой примеров показывает, как универсальный маршрут проектирования и разработки применяется в разных областях цифровых систем.

Документ не заменяет roadmap-документы, анкеты и энциклопедию. Он помогает выбрать подходящий пример для обучения и практики.

## 3. Правила слоя примеров

### RULE-EX-001. Пример должен проходить маршрут разработки

Хороший пример должен показывать не только код, но и путь от идеи до проверки, эксплуатации и развития.

### RULE-EX-002. Пример должен иметь область применения

Нужно явно указывать, к какой области относится пример:

- скрипты автоматизации;
- GUI-приложения;
- web-системы;
- embedded-системы;
- PLC-системы;
- CNC/CAM-системы;
- базы данных;
- интеграционные системы.

### RULE-EX-003. Категории и примеры не должны смешиваться

Категория должна быть отдельным уровнем, а конкретные примеры должны быть вложены внутрь категории.

### RULE-EX-004. Пример должен быть учебным и прикладным

Пример должен объяснять мышление разработчика, а не только давать готовый результат.

### RULE-EX-005. Пример не должен нарушать маршрут

Нельзя показывать путь `идея → код`, если пример является обучающим. Даже простой пример должен показывать минимальное проектирование.

## 4. Категории примеров

## 4.1. Scripts / Скрипты автоматизации

Назначение категории: показать проектирование и разработку небольших утилит, которые автоматизируют обработку файлов, таблиц, текстов, отчётов и повторяющихся действий.

Примеры:

- [[docs/06_examples/Scripts/Python_File_Processing_Utility|Python File Processing Utility]]
  - Полный учебный пример Python-утилиты для обработки файлов.
  - Показывает маршрут от идеи до тестирования, эксплуатации, сопровождения и развития.

Будущие примеры:

- утилита анализа Excel-таблиц;
- утилита парсинга логов;
- утилита генерации отчётов;
- утилита пакетной обработки файлов.

## 4.2. GUI / Графические приложения

Назначение категории: показать проектирование настольных приложений, интерфейсов, визуальных редакторов и рабочих панелей.

Будущие примеры:

- простая GUI-утилита для обработки файлов;
- редактор шаблонов;
- приложение предпросмотра и экспорта;
- интерфейс оператора.

## 4.3. Web / Web-системы

Назначение категории: показать проектирование web-интерфейсов, API, серверной логики и многопользовательских систем.

Будущие примеры:

- простой REST API;
- web-панель мониторинга;
- личный кабинет;
- сервис обработки файлов через браузер.

## 4.4. Embedded / Встроенные системы

Назначение категории: показать проектирование систем для микроконтроллеров, датчиков, исполнительных механизмов и устройств с ограниченными ресурсами.

Будущие примеры:

- контроллер датчика;
- управление клапаном;
- система сигнализации;
- логика безопасного состояния.

## 4.5. PLC / Промышленная автоматизация

Назначение категории: показать проектирование PLC-систем, HMI, аварий, режимов manual/auto и промышленной логики.

Будущие примеры:

- управление насосом;
- аварийная межблокировка;
- HMI-сценарий оператора;
- технологический автоматический режим.

## 4.6. CNC_CAM / CNC и CAM-системы

Назначение категории: показать проектирование систем для NC-программ, CAM-данных, постпроцессоров, анализа инструмента и производственных отчётов.

Будущие примеры:

- анализ NC-программы;
- подсчёт вызовов инструмента;
- отчёт по времени обработки;
- проверка постпроцессорного результата.

## 4.7. Databases / Базы данных

Назначение категории: показать проектирование хранения, моделей данных, миграций, журналов изменений и целостности данных.

Будущие примеры:

- складской учёт;
- журнал измерений;
- справочник инструментов;
- история изменений.

## 4.8. Integrations / Интеграционные системы

Назначение категории: показать обмен между системами, файлами, API, базами данных, PLC, GUI и внешними сервисами.

Будущие примеры:

- Excel → база данных;
- GUI → REST API;
- PLC → HMI;
- локальная утилита → отчётная система.

## 5. Общая структура учебного примера

Каждый полный пример должен иметь структуру:

```text
1. Назначение примера
2. Тип системы
3. Идея
4. Предметная область
5. Проектирование системы
6. Проектирование архитектуры системы
7. Технические требования
8. Связь требований с инструментарием
9. Выбор инструментария
10. Архитектура реализации
11. Кодовая структура
12. Тестирование
13. Эксплуатация
14. Сопровождение
15. Развитие системы
16. Выводы
```

## 6. Как пользоваться примерами

Пользователь может читать пример двумя способами:

### 6.1. Как учебный маршрут

Читать пример последовательно от идеи до развития системы.

### 6.2. Как справочник

Переходить к нужному разделу:

- требования;
- архитектура;
- инструментарий;
- тесты;
- эксплуатация;
- сопровождение;
- развитие.

## 7. Критерии хорошего примера

Пример считается качественным, если:

- показывает путь от идеи до результата;
- не перескакивает сразу к коду;
- отделяет проектирование системы от архитектуры системы;
- отделяет требования от выбора инструментов;
- показывает архитектуру реализации;
- показывает тестирование;
- показывает эксплуатацию;
- показывает сопровождение;
- показывает развитие системы;
- содержит минимум одну диаграмму Mermaid, если она помогает пониманию;
- не смешивает категории и примеры на одном уровне;
- содержит Obsidian wikilinks на ключевые документы маршрута.

# Регламент системы документации цифровых проектов

## 1. Назначение регламента

Регламент определяет правила построения системы документации проекта Programming Digital Systems.

Регламент применяется ко всем документам проекта: roadmap-документам, анкетам, энциклопедическим статьям, учебным главам, техническим требованиям, архитектурным описаниям, шаблонам и примерам.

## 2. Область применения

Регламент применяется при создании документов для следующих областей цифровых систем:

- скрипты автоматизации;
- GUI-приложения;
- web-системы;
- embedded-системы;
- PLC-системы;
- CNC/CAM-системы;
- базы данных;
- интеграционные системы.

## 3. Главный принцип

Документация должна быть не набором отдельных файлов, а связанной системой знаний.

Каждый документ должен показывать:

- откуда пришла информация;
- какое решение из неё следует;
- в какой следующий документ она передаётся;
- как пользователь должен двигаться дальше.

## 4. Типы документов

В проекте используются следующие типы документов:

### 4.1. Регламент

Определяет правила работы с документацией, структурой, ссылками, диаграммами и стилем изложения.

### 4.2. Roadmap-документ

Ведёт пользователя по проектному этапу и фиксирует порядок проектирования.

### 4.3. Анкета

Преобразует правила roadmap-документа в последовательность вопросов.

### 4.4. Энциклопедическая статья

Раскрывает отдельное понятие цифрового мира.

### 4.5. Пример

Показывает применение правила в конкретной области цифровых систем.

### 4.6. Диаграмма

Визуально отражает структуру, поток, состояние, последовательность или взаимосвязь.

## 5. Правило маршрута

Каждый roadmap-документ должен иметь:

- входные условия;
- порядок заполнения;
- контрольные вопросы;
- критерии завершения;
- выходные данные для следующего документа.

## 6. Правило связности

Каждый документ должен иметь раздел `Связанные документы`, если он влияет на другие документы или зависит от них.

Связь должна объяснять не только факт ссылки, но и смысл связи.

## 7. Правило визуальной информативности

Если информация описывает структуру, связь, поток, состояние, последовательность или иерархию, документ должен содержать визуальное представление.

Тип диаграммы должен соответствовать типу информации.

## 8. Правило анкет

Для каждого важного проектного этапа должна существовать связанная анкета.

Анкета должна обеспечивать движение от неопределённой идеи к зафиксированному проектному решению.

## 9. Правило масштабируемости

Документы должны быть пригодны для роста проекта до уровня книги, серии книг, энциклопедии и полной базы знаний.

Нельзя писать документы так, будто они существуют только для одной локальной задачи.

## 10. Связанные документы

- `PROJECT_SCOPE.md` — фиксирует масштаб и назначение проекта.
- `docs/01_regulations/Document_Writing_Rules.md` — определяет стиль и правила изложения.
- `docs/01_regulations/Link_Rules.md` — определяет правила ссылок.
- `docs/01_regulations/Diagram_Rules.md` — определяет правила диаграмм.
- `docs/02_templates/Roadmap_Document_Template.md` — задаёт шаблон roadmap-документа.
- `docs/02_templates/Questionnaire_Document_Template.md` — задаёт шаблон анкеты.

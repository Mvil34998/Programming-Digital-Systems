# Правила изложения технических документов

## 1. Назначение документа

Документ определяет правила написания технических, учебных и проектных материалов проекта Programming Digital Systems.

Цель правил — исключить личный шум, декоративный текст, нестрогие советы и смешивание разных уровней информации.

## 2. Запрещённые формы изложения

В документах запрещены:

- личные рассуждения;
- эмоциональные оценки;
- декоративные вступления;
- разговорные формулировки;
- нестрогие советы без условия применения;
- повторение одной мысли разными словами;
- формулировки без критерия проверки;
- смешивание требований, инструментов и реализации;
- смешивание категорий и примеров на одном уровне.

## 3. Обязательные формы изложения

Документ должен использовать проверяемые формулировки.

Допустимые формы:

- `Система должна...`
- `Разработчик должен...`
- `Документ должен...`
- `Требование считается выполненным, если...`
- `Если ..., то ...`
- `Запрещено ...`
- `Допускается ..., если ...`
- `Необходимо определить ...`
- `Перед переходом к следующему этапу должно быть зафиксировано ...`

## 4. Правило проверяемости

Каждое требование должно быть проверяемым.

Неправильно:

> Система должна быстро работать.

Правильно:

> Система должна обрабатывать входной файл объёмом до 5000 строк без зависания интерфейса.

## 5. Правило отделения требований от инструментов

Технические требования должны описывать свойства системы, а не способ реализации.

Неправильно:

> Использовать SQLite.

Правильно:

> Система должна хранить данные между запусками в структурированном виде.

Выбор SQLite, PostgreSQL, JSON или другого способа хранения выполняется в Roadmap выбора инструментария.

## 6. Правило отделения категорий от примеров

Категории и примеры нельзя смешивать на одном уровне.

Неправильно:

- Входные данные
- Excel-файл
- PDF-файл
- Конфигурационные данные

Правильно:

- Входные данные
  - Примеры:
    - Excel-файл
    - PDF-файл
- Конфигурационные данные
  - Примеры:
    - JSON-файл настроек
    - `.env`-файл

## 7. Правило примеров из разных областей

Если правило является универсальным, примеры должны показывать его применение в разных областях цифровых систем.

Области примеров:

- скрипт автоматизации;
- GUI-приложение;
- web-система;
- embedded-система;
- PLC-система;
- CNC/CAM-система;
- база данных;
- интеграционная система.

## 8. Правило учебного объяснения

Документ должен обучать образу мышления, но не должен превращаться в свободное эссе.

Учебное объяснение допускается, если оно:

- объясняет причину правила;
- показывает ошибку мышления;
- даёт проверяемый пример;
- ведёт пользователя к следующему проектному действию.

## 9. Правило завершения документа

Документ считается готовым, если:

- есть назначение;
- определена область применения;
- указаны связанные документы;
- структура не смешивает уровни информации;
- требования проверяемы;
- примеры вложены в категории;
- есть критерии завершения;
- есть следующий шаг.

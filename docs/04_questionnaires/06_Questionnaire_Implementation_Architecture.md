# Questionnaire: Implementation Architecture / Анкета архитектуры реализации

## 1. Назначение анкеты

`06_Questionnaire_Implementation_Architecture.md` превращает [[docs/03_roadmaps/06_Roadmap_Implementation_Architecture|Roadmap: Implementation Architecture]] в последовательность вопросов для проектирования конкретной структуры реализации цифровой системы.

Анкета должна помочь пользователю определить:

- структуру проекта;
- директории;
- модули реализации;
- точки входа;
- адаптеры;
- конфигурацию;
- обработку ошибок и логирование;
- тестовую структуру;
- сборку, запуск и поставку;
- правила зависимостей;
- открытые вопросы.

Анкета не должна писать код и не должна заново выбирать инструментарий.

## 2. Связанный roadmap-документ

- [[docs/03_roadmaps/06_Roadmap_Implementation_Architecture|Roadmap: Implementation Architecture]]
  - Передаёт: порядок проектирования архитектуры реализации.
  - Используется для: построения вопросов анкеты.
  - Ограничение: анкета фиксирует ответы пользователя, а не заменяет roadmap-документ.

## 3. Правила заполнения

- Каждый ответ должен быть конкретным.
- Каждая директория должна иметь назначение.
- Каждый модуль реализации должен иметь архитектурный источник.
- Каждая точка входа должна быть явно названа.
- Каждый адаптер должен закрывать внешнюю зависимость.
- Конфигурация должна иметь место хранения и правило проверки.
- Ошибки должны иметь маршрут обработки.
- Тестовая структура должна быть спроектирована до кодирования.
- Нельзя менять требования, архитектуру системы или инструментарий внутри этой анкеты.
- Если обнаружено противоречие, нужно создать открытый вопрос и вернуться к соответствующему этапу.

## 4. Входные данные для анкеты

Перед заполнением анкеты должны быть определены:

- архитектура системы;
- технические требования;
- выбранный инструментарий;
- ограничения выбранных инструментов;
- требования к тестируемости;
- требования к эксплуатации;
- требования к сопровождению;
- требования к документации.

Основные входные документы:

- [[docs/04_questionnaires/03_Questionnaire_Technical_Requirements|Questionnaire: Technical Requirements]].

## 5. Вопросы анкеты

## Q-IMPL-001. Какая структура проекта нужна системе?

### Вопрос

Как должно выглядеть дерево проекта на уровне директорий и основных файлов?

### Пояснение

Структура проекта должна отражать архитектуру системы, выбранный инструментарий и требования к тестированию, эксплуатации и сопровождению.

### Ответ

```text
project_root/
|-- 
|-- 
|-- 
```

### Критерий заполнения

Ответ считается заполненным, если указаны:

- корневая структура проекта;
- основные директории;
- основные технические файлы;
- назначение каждой части;
- связь структуры с архитектурой системы.

### Связанные документы

- [[docs/03_roadmaps/06_Roadmap_Implementation_Architecture#7.1. Структура директорий|Roadmap: Implementation Architecture — структура директорий]].

### Следующий шаг

Перейти к назначению директорий.

## Q-IMPL-002. Какое назначение имеет каждая директория?

### Вопрос

За что отвечает каждая директория проекта?

### Пояснение

Директория не должна существовать без назначения. Назначение директории должно помогать разработчику понимать, куда помещать новый код или артефакт.

### Ответ

```text
Директория:
Назначение:
Что можно хранить:
Что нельзя хранить:
Связанный архитектурный слой или модуль:

Директория:
Назначение:
Что можно хранить:
Что нельзя хранить:
Связанный архитектурный слой или модуль:
```

### Критерий заполнения

Ответ считается заполненным, если для каждой директории указаны:

- назначение;
- допустимое содержимое;
- запрещённое содержимое;
- связь с архитектурой системы или выбранным инструментарием.

### Связанные документы

- [[docs/03_roadmaps/06_Roadmap_Implementation_Architecture#10.2. Шаг 2. Определить структуру проекта|Шаг 2. Определить структуру проекта]].

### Следующий шаг

Перейти к модулям реализации.

## Q-IMPL-003. Какие модули реализации нужны системе?

### Вопрос

Какие конкретные модули кода должны реализовать архитектурные модули системы?

### Пояснение

Модуль реализации должен иметь архитектурный источник. Нельзя создавать модуль только потому, что удобно временно положить туда код.

### Ответ

```text
MODULE-IMPL-001:
Название:
Архитектурный источник:
Назначение:
Входные данные:
Выходные данные:
Зависимости:
Ошибки:
Тестирование:
Ограничения:

MODULE-IMPL-002:
Название:
Архитектурный источник:
Назначение:
Входные данные:
Выходные данные:
Зависимости:
Ошибки:
Тестирование:
Ограничения:
```

### Критерий заполнения

Ответ считается заполненным, если для каждого модуля указаны:

- название;
- архитектурный источник;
- назначение;
- входные и выходные данные;
- зависимости;
- ошибки;
- способ тестирования.

### Связанные документы

- [[docs/03_roadmaps/06_Roadmap_Implementation_Architecture#7.2. Модули кода|Roadmap: Implementation Architecture — модули кода]].

## Q-IMPL-004. Как архитектурные слои отражаются в реализации?

### Вопрос

Как слои архитектуры системы будут отражены в директориях, пакетах, модулях или проектах реализации?

### Пояснение

Не всегда слой обязан быть отдельной директорией, но правило отображения слоя должно быть явным.

### Ответ

```text
Архитектурный слой:
Реализация в проекте:
Директория или модуль:
Разрешённые зависимости:
Запрещённые зависимости:

Архитектурный слой:
Реализация в проекте:
Директория или модуль:
Разрешённые зависимости:
Запрещённые зависимости:
```

### Критерий заполнения

Ответ считается заполненным, если для каждого важного слоя указаны:

- где он находится в реализации;
- какие модули к нему относятся;
- какие зависимости разрешены;
- какие зависимости запрещены.

### Связанные документы

## Q-IMPL-005. Какие точки входа нужны системе?

### Вопрос

Откуда запускается система, сценарии, тесты, служебные команды или процессы?

### Пояснение

Точка входа должна быть понятной и документированной. Пользователь или разработчик должен понимать, какую команду запускать.

### Ответ

```text
Точка входа:
Назначение:
Файл или команда:
Кто запускает:
Входные параметры:
Выходной результат:
Ошибки запуска:

Точка входа:
Назначение:
Файл или команда:
Кто запускает:
Входные параметры:
Выходной результат:
Ошибки запуска:
```

### Критерий заполнения

Ответ считается заполненным, если для каждой точки входа указаны:

- назначение;
- файл, команда или блок запуска;
- кто или что запускает;
- входные параметры;
- результат;
- ошибки запуска.

### Связанные документы

- [[docs/03_roadmaps/06_Roadmap_Implementation_Architecture#6.3. Точка входа|Roadmap: Implementation Architecture — точка входа]].

## Q-IMPL-006. Где находятся модели реализации?

### Вопрос

Где будут расположены доменные модели, модели данных, DTO, модели событий, состояний, ошибок и конфигурации?

### Пояснение

Модели должны быть отделены от случайного procedural-кода и должны иметь понятную роль.

### Ответ

```text
Доменные модели:

Модели данных:

DTO / структуры обмена:

Модели конфигурации:

Модели ошибок:

Модели событий:

Модели состояний:

Модели хранения:

```

### Критерий заполнения

Ответ считается заполненным, если для каждой категории моделей указаны:

- место в проекте;
- назначение;
- связанные сущности или данные;
- кто использует модель;
- ограничения модели.

### Связанные документы

- [[docs/03_roadmaps/06_Roadmap_Implementation_Architecture#7.4. Реализация моделей|Roadmap: Implementation Architecture — реализация моделей]].

## Q-IMPL-007. Какие интерфейсы и адаптеры нужны реализации?

### Вопрос

Какие внешние зависимости должны быть закрыты интерфейсами или адаптерами?

### Пояснение

Адаптер нужен там, где внутренняя система взаимодействует с внешним инструментом, файлом, базой данных, API, UI, оборудованием или протоколом.

### Ответ

```text
Адаптер:
Внешняя зависимость:
Что скрывает:
Входные данные:
Выходные данные:
Ошибки:
Место в проекте:
Тестовая замена:

Адаптер:
Внешняя зависимость:
Что скрывает:
Входные данные:
Выходные данные:
Ошибки:
Место в проекте:
Тестовая замена:
```

### Критерий заполнения

Ответ считается заполненным, если для каждого адаптера указаны:

- внешняя зависимость;
- что адаптер скрывает;
- место в проекте;
- ошибки взаимодействия;
- способ тестовой замены.

## Q-IMPL-008. Как будет организована конфигурация реализации?

### Вопрос

Где хранятся настройки, как они загружаются, проверяются и используются?

### Пояснение

Конфигурация не должна быть размазана по коду. Если параметр должен меняться без изменения кода, он должен иметь явное место хранения.

### Ответ

```text
Конфигурационные файлы:

Место хранения:

Формат:

Кто загружает:

Кто проверяет:

Значения по умолчанию:

Ошибки конфигурации:

Действие при ошибке:

```

### Критерий заполнения

Ответ считается заполненным, если указаны:

- место хранения конфигурации;
- формат;
- загрузчик конфигурации;
- правила проверки;
- значения по умолчанию;
- обработка ошибок конфигурации.

### Связанные документы

- [[docs/03_roadmaps/06_Roadmap_Implementation_Architecture#7.6. Конфигурация реализации|Roadmap: Implementation Architecture — конфигурация реализации]].

## Q-IMPL-009. Как реализуются ошибки и логирование?

### Вопрос

Как ошибка проходит путь от возникновения до сообщения, лога, восстановления или остановки?

### Пояснение

Ошибки должны иметь маршрут обработки. Логирование должно помогать диагностике, а не быть случайным выводом текста.

### Ответ

```text
Где возникают ошибки:

Где ошибки перехватываются:

Где ошибки преобразуются в сообщения:

Где ошибки логируются:

Уровни логирования:

Формат лога:

Место хранения лога:

Правила диагностики:

Действие при критической ошибке:

```

### Критерий заполнения

Ответ считается заполненным, если указаны:

- маршрут ошибки;
- место логирования;
- уровни логирования;
- формат лога;
- действие при критической ошибке;
- связь с требованиями к диагностике.

Тестовые данные:

Fixtures:

Mock / fake-реализации:

Команда запуска тестов:

- место тестов;
- виды тестов;
- тестовые данные;
- fake/mock-реализации;
- команда запуска;
- связь с требованиями к тестируемости.

## Q-IMPL-011. Как система собирается, запускается и поставляется?

Какие команды, скрипты или процедуры нужны для запуска, проверки, сборки, установки и обновления системы?

Команды должны быть воспроизводимыми и понятными. Если запуск зависит от окружения, это должно быть указано.

Команда проверки качества:

Команда сборки:

Команда установки:

Команда обновления:

Ответ считается заполненным, если указаны:

- команды запуска;
- команды тестирования;
- команды сборки;
- способ поставки;
- ограничения окружения;
- связь с эксплуатацией.

## Q-IMPL-012. Какие зависимости разрешены и запрещены в реализации?

Какие модули могут зависеть друг от друга, а какие зависимости запрещены?

### Пояснение

Правила зависимостей защищают архитектуру от хаотичного кода и циклических импортов.

### Ответ

```text

Запрещённые зависимости:

Зависимости от внешних библиотек:

Где допустима инфраструктура:

Где инфраструктура запрещена:

Правила предотвращения циклических зависимостей:

Правила импортов:

```

### Критерий заполнения

Ответ считается заполненным, если указаны:

- разрешённые зависимости;
- запрещённые зависимости;
- правила внешних библиотек;
- правила инфраструктурных зависимостей;
- способ контроля циклических зависимостей.

## 6. Контроль полноты ответов

Анкета считается заполненной достаточно для перехода к кодированию и тестированию, если:

- определена структура проекта;
- для каждой директории указано назначение;
- определены модули реализации;
- определены точки входа;
- определено отображение архитектурных слоёв в реализации;
- определены модели реализации;
- определены интерфейсы и адаптеры;
- определена конфигурация;
- определены ошибки и логирование;
- определена тестовая структура;
- определены команды запуска, проверки и сборки;
- определены правила зависимостей;
- ограничения выбранных инструментов учтены;
- открытые вопросы вынесены отдельно.

## 7. Выходные данные анкеты

После заполнения анкеты должны быть получены:

- дерево проекта;
- описание директорий;
- список модулей реализации;
- список точек входа;
- отображение слоёв архитектуры системы в реализации;
- список моделей реализации;
- список интерфейсов и адаптеров;
- описание конфигурации;
- описание ошибок и логирования;
- тестовая структура;
- команды запуска, проверки и сборки;
- правила зависимостей;
- список открытых вопросов;
- входные данные для [[docs/04_questionnaires/07_Questionnaire_Testing|Questionnaire: Testing]];
- входные данные для [[docs/04_questionnaires/08_Questionnaire_Operation|Questionnaire: Operation]].

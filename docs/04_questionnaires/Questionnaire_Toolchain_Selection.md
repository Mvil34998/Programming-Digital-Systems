# Questionnaire: Toolchain Selection / Анкета выбора инструментария

## 1. Назначение анкеты

`Questionnaire_Toolchain_Selection.md` превращает `Roadmap_Toolchain_Selection.md` в последовательность вопросов для выбора инструментария конкретной цифровой системы.

Анкета должна помочь пользователю последовательно определить:

- категории нужных инструментов;
- требования-источники;
- архитектурные ограничения;
- критерии выбора;
- кандидатов;
- сравнение кандидатов;
- выбранные инструменты;
- ограничения выбранных инструментов;
- влияние выбора на архитектуру реализации;
- открытые вопросы.

Анкета не должна формировать технические требования заново.

## 2. Связанный roadmap-документ

- `docs/03_roadmaps/Roadmap_Toolchain_Selection.md`
  - Передаёт: порядок выбора инструментария.
  - Используется для: построения вопросов анкеты.
  - Ограничение: анкета должна фиксировать ответы пользователя, а не заменять roadmap-документ.

## 3. Правила заполнения

- Каждый выбранный инструмент должен быть связан с требованием, архитектурным ограничением или внешним обязательным ограничением.
- Нельзя выбирать инструмент только потому, что он знаком или популярен.
- Для важных решений должны быть указаны кандидаты и причина выбора.
- Для выбранного инструмента должны быть указаны ограничения.
- Если инструмент задан внешне, он фиксируется как обязательное ограничение.
- Нельзя изменять технические требования внутри этой анкеты.
- Если выбор инструмента выявил проблему в требованиях, нужно создать открытый вопрос и вернуться к требованиям.

## 4. Входные данные для анкеты

Перед заполнением анкеты должны быть определены:

- технические требования;
- требования, влияющие на выбор инструментария;
- архитектурные ограничения;
- требования к окружению;
- требования к эксплуатации;
- требования к сопровождению;
- требования к тестируемости;
- внешние обязательные ограничения.

Основные входные документы:

- `docs/04_questionnaires/Questionnaire_Technical_Requirements.md`.
- `docs/04_questionnaires/Questionnaire_System_Architecture_Design.md`.

## Q-TOOLS-001. Какие категории инструментов нужны системе?

### Вопрос

Какие виды инструментов необходимы для реализации, проверки, эксплуатации и сопровождения системы?

### Пояснение

Не каждая система требует всех категорий инструментов. Категории должны вытекать из требований и архитектуры системы.

### Ответ

```text
Язык программирования:

Среда выполнения:

GUI-инструменты:

Web-инструменты:

Embedded-инструменты:

PLC-инструменты:

Хранение и базы данных:

Форматы файлов:

Библиотеки и фреймворки:

Протоколы обмена:

Инструменты тестирования:

Инструменты логирования и диагностики:

Инструменты документации и диаграмм:

Инструменты сборки и развёртывания:

Средства симуляции и отладки:

Контроль версий и автоматизация:

```

### Критерий заполнения

Ответ считается заполненным, если для каждой выбранной категории указаны:

- зачем категория нужна;
- какие требования её вызывают;
- какие архитектурные ограничения на неё влияют;
- является ли категория обязательной или дополнительной.

## 6. Контроль полноты ответов

Анкета считается заполненной достаточно для перехода к архитектуре реализации, если:

- все выбранные инструменты связаны с требованиями или архитектурными ограничениями;
- для важных решений указаны кандидаты;
- для важных решений указаны критерии выбора;
- для выбранных инструментов указаны причины выбора;
- для выбранных инструментов указаны ограничения;
- внешние обязательные ограничения отмечены отдельно;
- открытые вопросы вынесены отдельно;
- выбор инструментария не изменяет технические требования без возврата к ним.

## 7. Выходные данные анкеты

После заполнения анкеты должны быть получены:

- список выбранных языков программирования;
- список выбранных сред выполнения;
- список выбранных GUI, web, embedded или PLC-инструментов;
- список выбранных способов хранения;
- список выбранных форматов файлов;
- список выбранных библиотек и фреймворков;
- список выбранных протоколов обмена;
- список выбранных инструментов тестирования;
- список выбранных инструментов логирования и диагностики;
- список выбранных инструментов документации;
- список выбранных инструментов сборки и развёртывания;
- список ограничений выбранных инструментов;
- входные данные для архитектуры реализации.

## 8. Открытые вопросы

```text
Q-OPEN-001:

Q-OPEN-002:

Q-OPEN-003:

```

Открытый вопрос должен содержать:

- формулировку вопроса;
- почему он важен;
- какой выбор он блокирует;
- кто или что может дать ответ;
- нужно ли вернуться к техническим требованиям.

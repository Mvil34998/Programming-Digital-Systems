# Questionnaire: System Architecture Design / Анкета проектирования архитектуры системы

## 1. Назначение анкеты

`Questionnaire_System_Architecture_Design.md` превращает `Roadmap_System_Architecture_Design.md` в последовательность вопросов для проектирования архитектуры конкретной цифровой системы.

Анкета должна помочь пользователю последовательно определить:

- слои;
- модули;
- модели;
- интерфейсы;
- зависимости;
- конфигурации;
- точки расширения;
- архитектурные ограничения;
- открытые вопросы.

Анкета не должна выбирать конкретные библиотеки, фреймворки, базы данных, GUI-инструменты или структуру файлов реализации.

## 2. Связанный roadmap-документ

- `docs/03_roadmaps/Roadmap_System_Architecture_Design.md`
  - Передаёт: порядок проектирования архитектуры системы.
  - Используется для: построения вопросов анкеты.
  - Ограничение: анкета должна фиксировать ответы пользователя, а не заменять roadmap-документ.

## 3. Правила заполнения

- Каждый ответ должен быть конкретным.
- Неизвестные ответы должны выноситься в раздел `Открытые вопросы`.
- Нельзя выбирать конкретные инструменты реализации.
- Нельзя смешивать архитектуру системы с архитектурой реализации.
- Нельзя создавать слои, модули или точки расширения без назначения.
- Нельзя смешивать категории и примеры на одном уровне.
- Ответ считается корректным, если его можно использовать для технических требований, выбора инструментария или архитектуры реализации.

## 4. Входные данные для анкеты

Перед заполнением анкеты должны быть определены:

- граница системы;
- внешние участники;
- сущности;
- данные;
- правила;
- состояния;
- события;
- потоки;
- хранение;
- ошибки;
- открытые вопросы проектирования системы.

Основной входной документ:

- `docs/04_questionnaires/Questionnaire_System_Design.md`.

## 5. Вопросы анкеты

## Q-SAD-001. Какие архитектурные цели должна поддерживать система?

### Вопрос

Какие архитектурные свойства должны быть обеспечены системой?

### Пояснение

Архитектурные цели должны вытекать из проектирования системы, а не из личных предпочтений по инструментам.

### Ответ

```text
Поддерживаемость:

Расширяемость:

Тестируемость:

Надёжность:

Разделение ответственности:

Интеграция:

Диагностика:

Другие цели:

```

### Критерий заполнения

Ответ считается заполненным, если для каждой архитектурной цели указано:

- зачем она нужна;
- какая часть системы на неё влияет;
- какой риск возникает без неё.

### Связанные документы

- `docs/03_roadmaps/Roadmap_System_Architecture_Design.md#1. Назначение документа`.
- `docs/05_encyclopedia/Architecture.md`.

## Q-SAD-002. Какие слои нужны системе?

### Вопрос

Какие архитектурные слои должны быть выделены в системе?

### Пояснение

Слой должен иметь область ответственности. Нельзя создавать слой только ради формальной структуры.

### Ответ

```text
Слой представления:

Слой сценариев / приложения:

Слой бизнес-логики:

Слой доменной модели:

Слой данных:

Слой хранения:

Слой инфраструктуры:

Слой конфигурации:

Слой ошибок и логирования:

Другие слои:

```

### Критерий заполнения

Ответ считается заполненным, если для каждого выбранного слоя указаны:

- назначение;
- ответственность;
- что слой получает на вход;
- что слой отдаёт на выход;
- допустимые зависимости;
- запрещённые зависимости.

### Связанные документы

- `docs/03_roadmaps/Roadmap_System_Architecture_Design.md#7. Слои`.
- `docs/05_encyclopedia/Architecture.md#7.1. Слои`.

```text
Модули конфигурации:
```

Модель должна формализовать смысловой объект системы. Модель не должна смешивать разные уровни ответственности.

### Ответ

```text
Доменные модели:

Модели данных:

Модели входных данных:

Модели выходных данных:

Модели промежуточных данных:

Модели конфигурации:

Модели состояния:

Модели событий:

Модели правил:

Модели ошибок:

Модели хранения:

Модели интерфейса:

Модели сценариев:

Модели связей:
```

- назначение;
- поля или состав;
- связанные сущности;
- связанные данные;
- ограничения;
- связи с другими моделями.

```text
Интерфейсы конфигурации:

Интерфейсы внешних систем:

Интерфейсы команд:

Интерфейсы тестирования:
```

## Q-SAD-006. Какие зависимости существуют в системе?

### Вопрос

Какие зависимости существуют между слоями, модулями, моделями, данными, конфигурацией, хранением, инфраструктурой, интерфейсами, событиями и окружением?

### Пояснение

Зависимости должны быть явными. Скрытые и циклические зависимости должны быть устранены или обоснованы.

### Ответ

```text
Внутренние зависимости:

Внешние зависимости:

Зависимости между слоями:

Зависимости между модулями:

Зависимости от моделей:

Зависимости от данных:

Зависимости от конфигурации:

Зависимости от хранения:

Зависимости от инфраструктуры:

Зависимости от интерфейсов:

Зависимости от событий:

Зависимости от времени и состояния:

Зависимости от окружения:

Тестовые зависимости:

Скрытые зависимости:

Циклические зависимости:

```

### Критерий заполнения

Ответ считается заполненным, если для каждой важной зависимости указаны:

- кто зависит;
- от чего зависит;
- зачем зависимость нужна;
- допустима ли зависимость;
- как зависимость влияет на тестирование;
- есть ли риск циклической или скрытой зависимости.

### Ответ

```text

Конфигурация путей:

Конфигурация источников данных:

Конфигурация выходных данных:

Конфигурация бизнес-параметров:

Конфигурация валидации:

Конфигурация интерфейса:

Конфигурация хранения:

Конфигурация логирования:

Конфигурация внешних систем:

Конфигурация окружения:

Конфигурация безопасности:

Конфигурация тестирования:

Конфигурация по умолчанию:

Пользовательская конфигурация:
```

Ответ считается заполненным, если для каждой важной конфигурации указаны:

- назначение;

### Пояснение

Точка расширения должна быть связана с реальным сценарием развития. Нельзя добавлять точки расширения без причины.

### Ответ

```text
Точки расширения входных данных:

Точки расширения парсинга:

Точки расширения бизнес-правил:

Точки расширения расчётов:

Точки расширения валидации:

Точки расширения моделей:

Точки расширения хранения:

Точки расширения выходных данных:

Точки расширения интерфейса пользователя:

Точки расширения внешних интеграций:

Точки расширения конфигурации:

Точки расширения ошибок и логирования:

Точки расширения событий:

Точки расширения команд:

Точки расширения тестирования:

Точки расширения плагинов:

```

### Критерий заполнения

Ответ считается заполненным, если для каждой точки расширения указаны:

- назначение;
- сценарий развития;
- архитектурная граница;
- какие элементы можно добавлять;
- какие элементы нельзя менять;
- риск усложнения.

Какие архитектурные решения должны ограничивать дальнейшие технические требования, выбор инструментария и реализацию?

### Пояснение

Архитектурное ограничение должно помогать сохранить структуру системы и предотвратить хаотичные решения на следующих этапах.

### Ответ

```text
Ограничения по слоям:

Ограничения по модулям:

Ограничения по моделям:

Ограничения по интерфейсам:

Ограничения по зависимостям:

Ограничения по конфигурации:

Ограничения по расширению:

Ограничения по ошибкам и логированию:

Ограничения по тестированию:
```

### Критерий заполнения

Ответ считается заполненным, если для каждого ограничения указаны:

- причина ограничения;
- на какие элементы системы оно влияет;
- какой риск оно предотвращает;
- какой следующий документ должен учитывать ограничение.

## 6. Контроль полноты ответов

Анкета считается заполненной достаточно для перехода к техническим требованиям, если:

- определены архитектурные цели;
- определены слои;
- определены модули;
- определены модели;
- определены интерфейсы;
- определены зависимости;
- определены конфигурации;
- определены точки расширения;
- определены архитектурные ограничения;
- архитектура системы не смешана с архитектурой реализации;
- ответы не содержат выбора конкретных инструментов;
- открытые вопросы вынесены отдельно;
- ответы можно использовать в `Roadmap_Technical_Requirements.md` и `Roadmap_Toolchain_Selection.md`.

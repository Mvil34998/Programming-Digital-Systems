# Questionnaire: Technical Requirements / Анкета технических требований

## 1. Назначение анкеты

`03_Questionnaire_Technical_Requirements.md` превращает [[docs/03_roadmaps/03_Roadmap_Technical_Requirements|Roadmap: Technical Requirements]] в последовательность вопросов для формирования технических требований конкретной цифровой системы.

Анкета должна помочь пользователю последовательно определить:

- требования к данным;
- требования к обработке;
- требования к хранению;
- требования к интерфейсам;
- требования к производительности;
- требования к надёжности;
- требования к ошибкам и диагностике;
- требования к конфигурации;
- требования к расширяемости;
- требования к тестируемости;
- требования к безопасности;
- требования к окружению;
- требования к эксплуатации;
- требования к сопровождению;
- требования к документации;
- требования, влияющие на следующие этапы;
- открытые вопросы.

Анкета не должна выбирать конкретные инструменты реализации.

## 2. Связанный roadmap-документ

- [[docs/03_roadmaps/03_Roadmap_Technical_Requirements|Roadmap: Technical Requirements]]
  - Передаёт: порядок формирования технических требований.
  - Используется для: построения вопросов анкеты.
  - Ограничение: анкета должна фиксировать ответы пользователя, а не заменять roadmap-документ.

## 3. Правила заполнения

- Каждый ответ должен быть конкретным.
- Каждое требование должно быть проверяемым.
- У каждого требования должен быть источник.
- У каждого требования должен быть критерий выполнения.
- У каждого требования должен быть способ проверки.
- Нельзя выбирать конкретные инструменты реализации.
- Нельзя смешивать технические требования с архитектурой реализации.
- Неизвестные ответы должны выноситься в раздел `Открытые вопросы`.
- Ответ считается корректным, если его можно использовать для проверки системы или для следующего проектного этапа.

## 4. Входные данные для анкеты

Перед заполнением анкеты должны быть определены:

- результаты проектирования системы;
- результаты проектирования архитектуры системы;
- граница системы;
- сущности;
- данные;
- правила;
- состояния;
- события;
- потоки;
- хранение;
- ошибки;
- слои;
- модули;
- модели;
- интерфейсы;
- зависимости;
- конфигурации;
- точки расширения;
- архитектурные ограничения.

Основные входные документы:

- [[docs/04_questionnaires/01_Questionnaire_System_Design|Questionnaire: System Design]].
- [[docs/04_questionnaires/02_Questionnaire_System_Architecture_Design|Questionnaire: System Architecture Design]].

## Q-TR-001. Какие требования к данным должна выполнять система?

### Вопрос

Какие данные система должна принимать, проверять, преобразовывать, хранить и выдавать?

### Пояснение

Требования к данным должны описывать сами данные и правила работы с ними, а не инструмент хранения или библиотеку обработки.

### Ответ

```text
Требования к входным данным:

Требования к внутренним данным:

Требования к хранимым данным:

Требования к выходным данным:

Требования к конфигурационным данным:

Требования к справочным данным:

Требования к событийным данным:

Требования к диагностическим данным:

```

### Критерий заполнения

Ответ считается заполненным, если для каждого важного требования указаны:

- формулировка через `Система должна...`;
- источник требования;
- критерий выполнения;
- способ проверки;
- действие при некорректных данных.

## Q-TR-016. Какие требования влияют на следующие этапы?

### Вопрос

Какие технические требования будут влиять на выбор инструментария, архитектуру реализации, тестирование, эксплуатацию и сопровождение?

### Ответ

```text
Требования, влияющие на выбор инструментария:

Требования, влияющие на архитектуру реализации:

Требования, влияющие на тестирование:

Требования, влияющие на эксплуатацию:

Требования, влияющие на сопровождение:

```

### Критерий заполнения

Ответ считается заполненным, если для каждого требования указаны идентификатор, следующий этап, причина влияния, ограничение и запрет для следующего этапа.

Перейти к контролю полноты ответов.

## 6. Контроль полноты ответов

Анкета считается заполненной достаточно для перехода к отдельному выбору инструментария, если:

- требования разделены по категориям;
- каждое важное требование имеет источник;
- каждое важное требование имеет критерий выполнения;
- каждое важное требование имеет способ проверки;
- требования не содержат выбора конкретных инструментов реализации;
- требования не противоречат архитектуре системы;
- определено влияние требований на следующие этапы;
- открытые вопросы вынесены отдельно;
- ответы можно использовать в [[docs/00_maps/04_Requirements_To_Toolchain_Map|Requirements To Toolchain Map]] и [[docs/03_roadmaps/05_Roadmap_Toolchain_Selection|Roadmap: Toolchain Selection]].

## 7. Выходные данные анкеты

После заполнения анкеты должны быть получены:

- список требований к данным;
- список требований к обработке;
- список требований к хранению;
- список требований к интерфейсам;
- список требований к производительности;
- список требований к надёжности;
- список требований к ошибкам и диагностике;
- список требований к конфигурации;
- список требований к расширяемости;
- список требований к тестируемости;
- список требований к безопасности;
- список требований к окружению;
- список требований к эксплуатации;
- список требований к сопровождению;
- список требований к документации;
- список требований, влияющих на выбор инструментария;
- список требований, влияющих на архитектуру реализации;
- список открытых вопросов;
- входные данные для [[docs/04_questionnaires/05_Questionnaire_Toolchain_Selection|Questionnaire: Toolchain Selection]];
- входные данные для [[docs/04_questionnaires/06_Questionnaire_Implementation_Architecture|Questionnaire: Implementation Architecture]].

## 8. Открытые вопросы

```text
Q-OPEN-001:

Q-OPEN-002:

Q-OPEN-003:

```

Открытый вопрос должен содержать:

- формулировку вопроса;
- почему он важен;
- какой этап он блокирует;
- кто или что может дать ответ;
- что нельзя делать до получения ответа.

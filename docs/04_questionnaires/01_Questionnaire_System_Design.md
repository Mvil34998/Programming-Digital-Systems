# Questionnaire: System Design / Анкета проектирования системы

## 1. Назначение анкеты

`01_01_Questionnaire_System_Design.md` превращает [[docs/03_roadmaps/01_01_Roadmap_System_Design|Roadmap: System Design]] в последовательность вопросов для проектирования конкретной цифровой системы.

Анкета должна помочь пользователю последовательно определить:

- границу системы;
- внешних участников;
- сущности;
- данные;
- правила;
- состояния;
- события;
- потоки;
- хранение;
- ошибки;
- открытые вопросы.

Анкета не должна выбирать инструменты реализации и не должна проектировать архитектуру системы.

## 2. Связанный roadmap-документ

- [[docs/03_roadmaps/01_01_Roadmap_System_Design|Roadmap: System Design]]
  - Передаёт: порядок проектирования системы.
  - Используется для: построения вопросов анкеты.
  - Ограничение: анкета должна фиксировать ответы пользователя, а не заменять roadmap-документ.

## 3. Правила заполнения

- Каждый ответ должен быть конкретным.
- Неизвестные ответы должны выноситься в раздел `Открытые вопросы`.
- Нельзя писать `потом решим`, если вопрос влияет на следующий этап.
- Нельзя выбирать библиотеки, базы данных, GUI-фреймворки и другие инструменты реализации.
- Нельзя смешивать категории и примеры на одном уровне.
- Ответ считается корректным, если его можно использовать для проектного решения или следующего этапа.

## 4. Входные данные для анкеты

Перед заполнением анкеты должны быть известны:

- идея системы;
- назначение системы;
- основная проблема;
- ожидаемый результат;
- основной пользователь или внешний потребитель результата;
- известные входные данные, файлы, команды, сигналы или источники.

## Q-SD-002. Где проходит граница системы?

### Вопрос

Что входит в систему и что находится вне системы?

### Пояснение

Необходимо отделить ответственность системы от пользователя, внешних программ, оборудования и ручных действий.

### Ответ

```text
Внутри системы:

Вне системы:

```

### Критерий заполнения

Ответ считается заполненным, если указаны:

- элементы внутри системы;
- элементы вне системы;
- внешние участники;
- внешние источники данных;
- внешние потребители результата.

## Q-SD-003. Какие сущности существуют в системе?

### Вопрос

Какие значимые объекты предметной области, данных, интерфейса, системы и результата должна учитывать система?

### Пояснение

Сущность должна иметь смысловую роль. Нельзя включать слово в список сущностей только потому, что оно встречается в описании проекта.

### Ответ

```text
Предметные сущности:

Информационные сущности:

Системные сущности:

Интерфейсные сущности:

Результирующие сущности:

```

### Критерий заполнения

Ответ считается заполненным, если для каждой важной сущности указаны:

- название;
- назначение;
- тип сущности;
- связь с другими сущностями;
- открытые вопросы, если они есть.

Какие данные система получает, создаёт, хранит, изменяет и выдаёт?

Данные должны быть связаны с сущностями, процессами, состояниями, событиями, хранением или результатами.

### Ответ

```text
Входные данные:

Внутренние данные:

Хранимые данные:

Выходные данные:

Конфигурационные данные:

Справочные данные:

Событийные данные:

Диагностические данные:

```

### Критерий заполнения

Ответ считается заполненным, если для каждого важного вида данных указаны:

- источник;
- формат;
- обязательность;
- связанная сущность;
- правило проверки;
- действие при ошибке.

## Q-SD-005. Какие правила управляют поведением системы?

### Вопрос

Какие правила определяют проверку, обработку, расчёты, доступ, переходы состояния, события и ошибки?

### Пояснение

Правило должно быть проверяемым. Нельзя писать общие фразы без условия применения и результата.

### Ответ

```text
Правила проверки данных:

Бизнес-правила:

Технические правила:

Правила обработки:

Правила расчётов:

Правила переходов состояния:

Правила событий:

Правила доступа:

Правила ошибок:

```

### Критерий заполнения

Ответ считается заполненным, если для каждого критичного правила указаны:

- условие применения;
- входные данные;
- результат применения;
- действие при нарушении;
- способ проверки.

## Q-SD-006. Какие состояния существуют в системе?

### Вопрос

В каких состояниях может находиться система, сущности, процессы, интерфейсы, оборудование, данные и ошибки?

### Пояснение

Состояние нужно фиксировать только если оно влияет на допустимые действия, правила, события, интерфейс, обработку или ошибки.

### Ответ

```text
Состояния системы:

Состояния сущностей:

Состояния процессов:

Состояния интерфейса:

Состояния оборудования:

Состояния данных:

Состояния ошибок:

```

### Критерий заполнения

Ответ считается заполненным, если для каждого важного состояния указаны:

- владелец состояния;
- смысл состояния;
- условия входа;
- условия выхода;
- допустимые действия;
- запрещённые действия;
- связанные события.

- источник;

- данные события;
- допустимое состояние;
- реакция системы;

## Q-SD-008. Какие потоки существуют в системе?

Как движутся данные, команды, события, состояния, ошибки и сообщения между частями системы?

### Пояснение

Поток должен иметь источник, получатель, передаваемый объект, причину запуска и порядок шагов.

### Ответ

```text

Потоки управления:

Пользовательские потоки:

Потоки событий:

Потоки состояний:

Потоки ошибок:

Потоки хранения:
```

### Критерий заполнения

Ответ считается заполненным, если для каждого критичного потока указаны:

- источник;
- получатель;
- что передаётся;
- что запускает поток;
- порядок шагов;
- возможные ошибки.

## Q-SD-009. Что система должна хранить?

### Вопрос

Какие данные должны сохраняться, зачем они сохраняются и каков их жизненный цикл?

### Пояснение

На этом этапе нельзя выбирать конкретную базу данных, файл или библиотеку. Нужно определить потребность в хранении.

### Ответ

```text
Данные для хранения:

Назначение хранения:

Кто создаёт данные:

Кто использует данные:

Жизненный цикл данных:

Правила обновления:

Правила архивирования или удаления:

Требования к целостности:

```

### Критерий заполнения

Ответ считается заполненным, если для каждого вида хранения указаны:

- что сохраняется;
- зачем сохраняется;
- кто создаёт данные;
- кто использует данные;
- срок жизни;
- правило обновления;
- возможные ошибки хранения.

## Q-SD-010. Какие ошибки возможны в системе?

### Вопрос

Какие ошибки могут возникнуть при входных данных, обработке, состояниях, хранении, интеграции, конфигурации, интерфейсе, безопасности или оборудовании?

### Пояснение

Ошибка должна иметь источник, причину, критичность, реакцию системы и способ проверки.

### Ответ

```text
Ошибки входных данных:

Ошибки обработки:

Ошибки состояния:

Ошибки хранения:

Ошибки интеграции:

Ошибки конфигурации:

Ошибки интерфейса:

Ошибки безопасности:

Аппаратные и промышленные ошибки:

```

### Критерий заполнения

Ответ считается заполненным, если для каждой критичной ошибки указаны:

- источник;
- причина;
- условие возникновения;
- уровень критичности;
- реакция системы;
- сообщение пользователю или оператору;
- правило логирования;
- способ проверки.

- определена граница системы;
- определены внешние участники;
- определены сущности;
- определены данные;
- определены правила;
- определены состояния;
- определены события;
- определены потоки;
- определено хранение;
- определены ошибки;
- открытые вопросы вынесены отдельно;
- ответы не содержат выбора инструментов реализации;
- ответы можно использовать в [[docs/03_roadmaps/02_02_Roadmap_System_Architecture_Design|Roadmap: System Architecture Design]].

# Questionnaire: Testing / Анкета тестирования

## 1. Назначение анкеты

`Questionnaire_Testing.md` превращает `Roadmap_Testing.md` в последовательность вопросов для проектирования и фиксации проверки конкретной цифровой системы.

Анкета должна помочь определить:

- какие требования проверяются;
- какие объекты тестируются;
- какие виды тестирования нужны;
- какие тестовые данные используются;
- какие сценарии выполняются;
- какие ожидаемые результаты должны быть получены;
- какие критерии приёмки применяются;
- какие проверки автоматические;
- какие проверки ручные;
- что делать при провале теста;
- какие результаты нужно зафиксировать.

Анкета не должна изменять требования, архитектуру или код без возврата к соответствующему этапу.

## 2. Связанный roadmap-документ

- `docs/03_roadmaps/Roadmap_Testing.md`
  - Передаёт: порядок проектирования тестирования.
  - Используется для: построения вопросов анкеты.
  - Ограничение: анкета фиксирует ответы пользователя, а не заменяет roadmap-документ.

## 3. Правила заполнения

- Каждый тест должен иметь источник.
- Каждый тест должен иметь ожидаемый результат.
- Каждый критичный сценарий должен иметь проверку.
- Ошибочные сценарии должны проверяться отдельно от успешных.
- Тестовые данные должны быть воспроизводимыми.
- Ручная проверка должна иметь сценарий.
- Провал теста должен приводить к анализу причины.
- Если тест выявляет проблему требования, нужно вернуться к требованиям.
- Если тест выявляет проблему архитектуры, нужно вернуться к архитектуре.
- Если тест выявляет проблему реализации, нужно создать задачу исправления.

## 4. Входные данные для анкеты

Перед заполнением анкеты должны быть определены:

- технические требования;
- архитектура реализации;
- структура проекта;
- модули реализации;
- интерфейсы;
- адаптеры;
- обработка ошибок;
- логирование;
- тестовая структура;
- команды запуска;
- критерии приёмки, если они уже заданы.

Основные входные документы:

- `docs/04_questionnaires/Questionnaire_Technical_Requirements.md`.
- `docs/04_questionnaires/Questionnaire_Implementation_Architecture.md`.

### Пояснение

### Ответ

```text

Формулировка:
Критерий выполнения:
Способ проверки:
Приоритет:

REQ-TR-002:
Формулировка:
Критерий выполнения:
Способ проверки:
Приоритет:

REQ-TR-003:
Формулировка:
```

- формулировка;
- критерий выполнения;
- способ проверки;
- приоритет проверки.

## 6. Контроль полноты ответов

Анкета считается заполненной достаточно для перехода к эксплуатации, если:

- критичные требования имеют тесты;
- критичные ошибки имеют тесты;
- определены объекты тестирования;
- определены виды тестирования;
- определены тестовые данные;
- успешные сценарии описаны;
- ошибочные сценарии описаны;
- интерфейсы проверяются;
- критерии производительности проверяются, если они есть;
- требования безопасности проверяются, если они есть;
- регрессионные проверки определены;
- ручные проверки имеют сценарии;
- действие при провале теста определено;
- результаты тестирования фиксируются;
- открытые вопросы вынесены отдельно.

- список ошибочных сценариев;

## 8. Открытые вопросы

```text
Q-OPEN-001:

Q-OPEN-002:

Q-OPEN-003:

```

Открытый вопрос должен содержать:

- формулировку вопроса;
- почему он важен;
- какой тест или этап он блокирует;
- кто или что может дать ответ;
- нужно ли вернуться к требованиям, архитектуре, инструментарию или реализации.
